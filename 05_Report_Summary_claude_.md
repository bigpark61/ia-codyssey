# 3) 섹션별 핵심 메시지

### 섹션 1. 보고 배경
> 고객문의 대응 업무의 구조적 비효율이 지속되고 있으며, 생성형 AI 기반 상담 지원 방식의 도입 가능성을 제한된 범위에서 검증할 필요가 있음. 본 보고서는 전면 도입 확정안이 아니라, **파일럿 착수 승인**을 요청하는 단계의 검토안임.

### 섹션 2. 고객문의 대응 현황 및 운영상 문제
> 월 18,000건 수준의 문의가 4개 채널에 분산되어 있으며, 반복 유형 문의 비중이 높아 상담 인력이 동일 내용을 반복 처리하는 구조가 고착되어 있음. 이로 인해 응답 속도 편차와 특정 시간대 문의 적체가 발생하고 있음. 기존 규칙 기반 챗봇은 고정 FAQ 응답만 가능하여 대응 범위에 한계가 있음.

### 섹션 3. 도입 대안 비교 검토
> 세 가지 대안(현행 유지 / 규칙 기반 챗봇 고도화 / 생성형 AI 상담 지원형 파일럿)을 비교 검토함. 각 대안은 비용 구조, 운영 유연성, 리스크 수준이 상이하며, 어느 방향이 적합한지를 판단하기 위해 **생성형 AI 방식의 파일럿 검토가 선행될 필요**가 있음. 이 단계에서 우열을 단정하지 않으며, 파일럿을 통한 실증 검토가 목적임.

### 섹션 4. 파일럿 추진 계획 및 예산 범위
> 파일럿 예산은 8천만 원 이내로 설정하며, 초기 대규모 집행 없이 단계적으로 집행함. 완전 자동응답 방식이 아닌 **상담사 보조형(Human-in-the-loop)** 방식을 우선 적용하여 운영 리스크를 최소화함. 예산은 인프라 이용료, 구축·연동 비용, 운영 지원 비용으로 구분하여 관리함.

### 섹션 5. 기대 효과 판단 기준
> 이 단계에서는 ROI, 비용 절감률 등 정량 효과를 추정하지 않음. 파일럿의 성공 여부는 다음 기준으로 판단함: ① 반복 문의 유형의 AI 보조 처리 가능 여부 확인, ② 상담사의 실사용 편의성 및 수용도, ③ 답변 정확성 및 오류 발생 수준, ④ 개인정보 처리 통제 방안의 실제 적용 가능성.

### 섹션 6. 리스크 및 통제 방안
> 주요 리스크는 개인정보 처리, 답변 품질 불안정, 상담사 업무 혼선의 세 가지임. 외부 API에 원문 개인정보를 직접 전송하는 방식은 금지 원칙을 전제로 하며, 마스킹·비식별 처리 방안을 파일럿 설계 단계에서 확정함. 답변은 상담사 검수 후 발송하는 구조로 운영하여 오응답 리스크를 통제함.

### 섹션 7. 파일럿 추진 일정
> 2026년 Q2 이전 착수를 목표로 하며, 준비 단계(FAQ 정비, 개인정보 처리 방안 확정, 벤더 선정)와 실행 단계(파일럿 운영, 모니터링, 결과 평가)로 구분함. 파일럿 종료 후 결과를 바탕으로 다음 단계 여부를 별도 보고함.

### 섹션 8. 승인 요청 사항
> 본 보고서는 전면 도입 확정이 아닌, **제한된 범위의 파일럿 착수**에 대한 승인을 요청함. 승인 항목은 ① 파일럿 추진 방향 승인, ② 예산 상한(8천만 원 이내) 집행 승인, ③ 파일럿 결과 보고 후 차기 단계 검토 원칙 확인의 세 가지임.

# 5) 리스크/반론 포인트

**① 개인정보 처리 리스크 (CFO·보안팀 관점)**
- 외부 API 연동 시 고객 민감정보(주문·주소·연락처) 노출 가능성
- 마스킹·비식별 처리 기술의 실제 적용 가능성이 파일럿 착수 전 미확정 상태
- → **대응 방향:** 원문 개인정보 직접 전송 금지를 설계 전제로 명시하고, 처리 방안은 파일럿 준비 단계에서 확정

**② 답변 품질 불안정 리스크 (COO·운영본부장 관점)**
- 생성형 AI 특성상 오응답(환각) 발생 가능성 존재
- 잘못된 자동 응답이 고객 불만 및 브랜드 신뢰 훼손으로 이어질 수 있음
- → **대응 방향:** 상담사 보조형 운영(검수 후 발송)으로 완전 자동응답 방식 배제

**③ 비용 대비 실효성 불확실 (CFO 관점)**
- 파일럿 단계에서 정량적 ROI 산출이 어려워 투자 타당성 판단 기준이 불명확할 수 있음
- → **대응 방향:** 정량 효과 추정 대신 파일럿 성공 판단 기준(4개 항목)을 사전에 명시하여 평가 기준 선제 설정

**④ 내부 준비 미흡 리스크**
- FAQ 문서 정비 수준의 부서별 편차가 크고, 반복 문의 공식 분류 체계 미정
- 생성형 AI 도입 경험 전무로 내부 운영 역량 불확실
- → **대응 방향:** 파일럿 범위를 FAQ 정비가 상대적으로 완료된 영역으로 한정하여 착수

# 6) 확인 필요 사항

> 아래 3개 항목은 보고서 작성 전 또는 파일럿 설계 단계에서 반드시 확인이 필요합니다.

**① 개인정보 마스킹·비식별 처리 방안의 기술적 실현 가능성**
- 외부 API 연동 전 원문 개인정보를 마스킹 또는 비식별 처리하는 방식이 현재 내부 시스템 구조에서 구현 가능한지 보안팀과 사전 협의 필요
- 처리 방안이 확정되지 않으면 파일럿 설계 자체가 불가

**② 반복 문의 유형 분류 및 파일럿 적용 범위 확정**
- 현재 반복 문의 비중은 내부 추정 수준이며 공식 분류 체계 미정
- 파일럿 대상 문의 유형과 채널을 어느 범위로 한정할지 운영팀과 사전 합의 필요

**③ FAQ 문서 정비 우선순위 및 착수 가능 시점**
- 부서별 FAQ 문서 편차가 크므로, 파일럿 착수 전 최소 기준을 충족하는 영역 특정 필요
- Q2 착수 일정 역산 시 FAQ 정비 완료 시점이 임계 경로(Critical Path)가 될 수 있으므로 조기 확인 요망
