# LLM 공통 테스트용 샘플 보고서

## 과업: 보고서 개요 생성

## 1. 문서 개요

본 문서는 3개의 LLM(GPT-4.1, Claude Sonnet 4.6, Gemini 2.5 Pro)에 동일하게 입력하여 **보고서 개요 생성 성능**을 비교하기 위해 작성한 테스트용 샘플 브리프이다.  
테스트 목적은 각 모델이 동일한 조건에서 다음 요소를 얼마나 잘 반영하는지 확인하는 것이다.

- 보고 목적에 맞는 구조 설계
- 대상 독자에 따른 강조점 조정
- 제약 조건 및 금지 사항 반영
- 리스크와 확인 필요 사항 구분
- 불필요한 수치 환각 억제

## 2. 테스트용 입력 문서(원문)

### [업무 과업]

보고서 개요 생성

### [보고서 주제]

생성형 AI 기반 고객문의 자동화 파일럿 도입 검토안

### [보고 목적]

2026년 상반기 내 파일럿 추진 여부에 대한 경영진 승인 검토를 위한 보고서 개요를 작성한다.  
이번 보고서는 최종 도입 확정안이 아니라, **제한된 범위의 파일럿 착수 승인**을 요청하는 목적을 가진다.

### [대상 독자]

- CFO
- COO
- 운영본부장

단, 이번 보고서는 **비용 통제와 리스크 관리 관점이 중요하므로 CFO 관점을 우선 반영**해야 한다.

### [보고서에서 전달하려는 핵심 메시지]

현재 고객문의 대응 업무는 반복 문의 비중이 높고 상담 인력 의존도가 크기 때문에, 생성형 AI 기반 자동화 도입 가능성을 검토할 필요가 있다.  
다만 이번 단계에서는 공격적인 ROI 추정보다, **운영 효율 개선 가능성 검토와 개인정보·품질 리스크 통제 가능성 확인**에 초점을 두어야 한다.  
따라서 전면 도입 제안이 아니라, **Q2 이전 제한 범위 파일럿 착수 승인안** 형태가 적절하다.

### [현재 상황 및 배경]

현재 회사의 고객문의 채널은 웹 문의, 이메일, 앱 내 문의, 콜센터로 분산되어 있다.  
전체 고객문의 중 자주 반복되는 유형이 많아 상담 인력이 같은 내용을 반복 안내하는 비중이 높다.  
그 결과 응답 속도 편차가 발생하고, 특정 시간대에는 문의 적체가 심해지고 있다.  
실무팀은 반복 문의 자동화, 상담사 보조, FAQ 추천 기능 등을 통해 운영 효율을 높일 수 있을 것으로 보고 있다.

다만 고객문의에는 주문 정보, 환불 요청, 주소, 연락처 등 민감 정보가 포함될 수 있어, 생성형 AI 활용 시 개인정보 처리 방식에 대한 우려가 존재한다.  
또한 답변 품질이 불안정할 경우 고객 불만과 CS 리스크가 커질 수 있으며, 잘못된 자동 응답은 브랜드 신뢰에 영향을 줄 수 있다.

### [내부 현황 메모]

- 현재 고객센터 운영 인력: 12명
- 월평균 고객문의: 약 18,000건
- 반복 문의 비중: 내부 추정상 높음, 다만 공식 분류 체계는 아직 미정
- FAQ 문서 정비 수준: 부서별 편차 큼
- 기존 챗봇: 규칙 기반 FAQ 응답만 가능
- 생성형 AI 도입 경험: 없음
- 보안팀 의견: 외부 API 사용 시 원문 개인정보 직접 전송은 금지 원칙 검토 필요
- 운영팀 의견: 완전 자동응답보다 상담사 보조형부터 시작하는 것이 현실적
- 재무팀 의견: 초기 도입비보다 운영 안정성과 통제 가능성을 먼저 확인해야 함

### [반드시 포함해야 할 섹션]

아래 항목은 보고서 개요에 반드시 들어가야 한다.

1. 보고 배경
2. 현재 운영상 문제
3. 도입 대안 비교
4. 예산 범위 및 집행 원칙
5. 기대 효과의 판단 기준
6. 리스크 및 통제 방안
7. 파일럿 추진 일정
8. 최종 승인 요청 사항

### [제약 조건]

- 파일럿 예산 상한은 **8천만 원 이내**
- **Q2 이전 착수**를 전제로 검토
- 외부 벤더 실명은 보고서에 넣지 않음
- 시장규모 통계나 업계 평균 수치를 임의로 넣지 않음
- ROI, 비용 절감률, 생산성 향상률 등 **정량 효과를 추정해서 쓰지 않음**
- 최종 도입 확정처럼 표현하지 말고, **파일럿 승인 검토안** 수준으로 표현
- 보고서 톤은 **임원 보고용으로 간결하고 단정하게** 유지

### [정책 및 금지 사항]

- 외부 API에 **원문 개인정보를 직접 전송하는 방식은 금지 전제**로 다룰 것
- 개인정보 마스킹 또는 비식별 처리 여부는 **확인 필요 항목**으로 남길 수 있음
- 검증되지 않은 효과 수치, 외부 기사 인용, 경쟁사 사례는 넣지 말 것
- 실무 부서의 기대감을 과장하지 말 것
- “전사 확대”, “즉시 전환”, “대규모 투자” 같은 표현은 사용하지 말 것

### [참고로 고려할 수 있는 대안]

다음 세 가지 대안을 비교하는 구조를 포함하면 좋다.

- 대안 1: 현행 유지
- 대안 2: 기존 규칙 기반 챗봇 고도화
- 대안 3: 생성형 AI 기반 상담 지원형 파일럿

단, 위 세 대안의 우열을 단정하기보다는, **파일럿 검토가 필요한 이유**가 드러나도록 구성하는 것이 좋다.

### [원하는 출력 형식]

최종 결과는 아래 형식으로 정리되길 원한다.

1. 보고서 제목
2. 권장 목차
3. 각 섹션별 핵심 메시지
4. 각 섹션에서 필요할 데이터 또는 근거
5. 예상 리스크 / 반론 포인트
6. 확인 필요 사항

### [확인 질문 규칙]

입력 정보가 부족하다고 판단되면 최대 3개까지 확인 질문을 먼저 할 수 있다.  
다만 현재 정보만으로도 개요 생성이 가능하다면 바로 작성한다.

## 3. LLM 입력용 최종 프롬프트 형태

아래는 실제로 3개 LLM에 공통 입력할 때 사용할 수 있는 테스트 프롬프트 형식이다.

```text
당신은 전략기획 실무자를 돕는 보고서 개요 생성 도우미다.
아래 브리프를 읽고, 경영진 보고용 보고서 개요를 작성하라.

규칙:
- 입력에 없는 사실, 수치, 정책은 지어내지 말 것
- 시장규모, ROI, 비용 절감률 등 정량 효과는 임의 추정 금지
- 외부 벤더 실명 금지
- 파일럿 승인 검토안 수준으로 작성
- 보고서 톤은 임원 보고용, 간결하고 단정하게 유지
- 정보가 부족하면 최대 3개까지 질문 가능
- 최종 출력은 아래 형식만 따를 것:
  1) 제목
  2) 권장 목차
  3) 섹션별 핵심 메시지
  4) 섹션별 필요 데이터/근거
  5) 리스크/반론 포인트
  6) 확인 필요 사항

[브리프]
(위 테스트용 입력 문서 전체 삽입)
```

## 4. 이 샘플 문서를 선택한 이유

이 테스트 샘플은 단순한 개요 생성이 아니라, 실제 업무 상황에서 자주 발생하는 난점을 포함하도록 설계했다.

첫째, **대상 독자가 CFO 중심**이므로 일반적인 기획 문서가 아니라 비용 통제와 승인 관점을 반영해야 한다.  
둘째, 예산 상한, 일정, 정책, 금지 표현 등 **제약 조건이 명확하게 포함**되어 있어 형식 준수와 입력 충실성을 평가하기 좋다.  
셋째, 개인정보 처리와 자동응답 품질 같은 **리스크 요소가 포함**되어 있어 환각 없이 보수적으로 구조화할 수 있는지 확인할 수 있다.  
넷째, 정량 효과를 임의 추정하지 말라는 조건이 있어, 모델이 그럴듯한 숫자를 만들어내는지 여부를 확인하기에 적합하다.

## 5. 평가 포인트 예시

이 샘플 보고서를 3개의 LLM에 입력했을 때 아래 항목을 중심으로 비교할 수 있다.

|평가 항목|확인 내용|
|---|---|
|형식 준수|제목, 목차, 핵심 메시지, 필요 데이터, 리스크, 확인 필요 사항을 모두 포함하는가|
|독자 맞춤성|CFO 중심 관점을 반영하는가|
|입력 충실성|예산 상한, Q2 이전 착수, 벤더명 금지 등 제약을 유지하는가|
|환각 억제|ROI나 시장규모 수치를 임의 생성하지 않는가|
|구조 논리성|보고 배경 → 문제 → 대안 → 비용/리스크 → 승인 요청 흐름이 자연스러운가|
|문맥 유지|추가 수정 요청 시 기존 조건을 유지하는가|
