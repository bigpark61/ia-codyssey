# Mission3 최종 산출물: 노코드 자동화 워크플로우 설계 및 구현

## [프로젝트 1] 자동화 도구 비교 구현 산출물

### 1. 사용한 도구 이름 및 공통 워크플로우 시나리오

- **사용 도구:** Make, Zapier
- **공통 시나리오:** Google Form 제출 → 점수 기준 조건 분기(80점 이상/미만) → Google Sheets 기록 + Slack/Discord 알림 전송,

### 2. 구현 과정 요약

**1) Make 구현 요약**

- **Trigger:** Google Forms 응답이 저장되는 Google Sheets를 `Watch New Rows` 모듈로 감시하여 새 행 추가를 감지합니다,.
- **조건 분기:** `Router` 모듈을 사용하여 점수(Score) 필드를 기준으로 `Score >= 80` (합격) 경로와 `Score < 80` (불합격) 경로로 분기합니다.
- **Action:** 각 경로에서 `Google Sheets (Add a Row)` 모듈을 통해 분리된 결과 기록용 시트에 'PASS' 또는 'FAIL' 판정 결과를 누적 기록한 후, `Slack (Send a Message)` 모듈로 담당자에게 최종 결과 알림을 자동 전송합니다,.

**2) Zapier 구현 요약**

- **Trigger:** Google Forms의 `New Form Response (Instant)` 트리거를 사용하여 폼 제출 즉시 Zap이 실행되도록 구성합니다,.
- **조건 분기:** `Paths by Zapier`를 사용하여 Path A(합격: `Score >= 80`)와 Path B(불합격: `Score < 80`) 두 가지 커스텀 규칙 경로로 나눕니다.
- **Action:** 각 Path 내부에서 `Google Sheets (Create Spreadsheet Row)` 액션을 통해 결과 시트에 데이터를 기록하고, `Slack (Send Channel Message)` 액션으로 합격/불합격 판정 메시지를 전송합니다,.

### 3. 비교 분석 보고서 (5개 항목)

1. **워크플로우 표현 방식 (UI/UX):**
    - **Make:** 시각적 캔버스 기반으로 모듈 간의 연결을 선으로 표현하며, Router를 통한 다중 분기 흐름을 한눈에 볼 수 있어 구조 설명에 유리합니다.
    - **Zapier:** 단계형 리스트 구조로 직관적이고 설정이 쉽지만, 분기가 많아질수록 복잡한 흐름을 시각적으로 파악하기 어렵습니다.
2. **실행 방식 (트리거 반응성):**
    - **Make:** Google Sheets 업데이트 주기를 감지(Polling)하는 스케줄(기본 15분) 기반으로 동작합니다.
    - **Zapier:** Google Form 제출 즉시 이벤트가 발생하는 'Instant' 트리거를 지원하여 실시간 반응성이 뛰어납니다.
3. **조건 분기 구조 (유연성):**
    - **Make:** `Router`를 통해 각 경로를 독립된 라인과 필터로 시각화하며, Fallback 라우트 설정 등 다중 흐름 설계에 매우 유연합니다.
    - **Zapier:** `Paths`를 이용해 "If A then X / If B then Y" 형태의 규칙을 리스트 형태로 구성하며, 순차적으로 경로를 평가합니다.
4. **사용 난이도:**
    - **Make:** 모듈, 라우터, 스케줄 등 전체 데이터 흐름을 이해해야 하므로 초기 학습 곡선이 있지만, 중급 이상에서는 설계 자유도가 높습니다.
    - **Zapier:** 초보자 친화적이며, 트리거 1개와 액션 몇 개를 순서대로 나열하는 방식이라 빠른 구축과 진입 장벽이 낮다는 장점이 있습니다.
5. **요금제 및 기능 접근성:**
    - **Make:** 운영 자원(실행 빈도, 파일 크기 등)과 크레딧을 기준으로 제약이 걸리며, 무료 플랜에서도 Router 등 복잡한 워크플로우 구성 기능을 사용할 수 있습니다,.
    - **Zapier:** Task 중심으로 비용을 책정하며, 2단계 이상의 다중 액션과 Paths 기능 같은 조건 분기는 무료 플랜에서 제한되어 유료 플랜(Professional 이상)이 요구됩니다.

### 4. 장단점 및 적합한 상황에 대한 의견

- **Make:** 워크플로우의 전체 구조를 시각적으로 보여주고 복잡한 분기 처리를 구현하기 좋아 **교육 목적이나 복잡한 시나리오를 설계하고 설명해야 하는 환경**에 가장 적합합니다.
- **Zapier:** 즉각적인 반응성과 빠른 세팅이 가능하여, **초보자가 빠르게 시도하거나 실무에서 즉시 동작하는 MVP 자동화를 구축해야 하는 환경**에 매우 적합합니다.

## [프로젝트 2] 자유 주제 자동화 설계 및 구현 산출물

### 1. 자동화할 반복 업무 정의

- **업무 정의:** 매일 여러 채널(이메일 등)로 들어오는 문의를 수작업으로 읽고 분류하여 각 담당 시트에 정리하고 팀 채널에 전달하는 업무.

  자동화 대상 업무 :
   온라인 시험을 가정하여 Google Form 을 통하여 이름, 이메일, 점수를 입력하면, 80점 이상이면 합격 메시지를, 80점 이하이면 불합격 메tl시지를 Discord Channel 로 알림 전송하는 업무.
    
    Google Form 제출 → 조건 분기(점수 기준) → Google Sheets 기록 + Discord 알림을 자동화
    
### 2. 선정 도구 및 선정 이유

- **선정 도구:** Make
- **선정 이유:** 문의 유형에 따른 다양한 조건 분기(환불, 버그, 일반 문의 등)가 필요하므로, 시각적 캔버스를 통해 조건별 흐름을 한눈에 설계하고 설명하기 좋은 Make가 적합합니다. 또한 `Router`를 통해 여러 처리 경로를 깔끔하게 분리할 수 있습니다.

### 3. 워크플로우 설계 문서

이 워크플로우는 이메일 문의가 수신되면, AI를 활용해 내용을 분석/요약하고, 문의 성격에 따라 자동으로 시트를 적재하며, 관련 담당자에게 메신저 알림을 보내는 구조입니다 (보너스 과제 요소 포함),.

**워크플로우 흐름:** MAKE/Zapier

1. **Trigger:** Google Form에서 새로운 Row 입력을 감지합니다.
2. **조건 분기 (Router):** SCORE(점수) 를 읽어서 80점 이상이면 Action1 으로 80점 이하이면 Axction3 로 분기합니다.
3. **Action 1 (합격데이터 기록):** 80점 이상이면  Processed_Results 탭에 내용을 입력합니다.
4. **Action2 (DIscord 알림)** 합격 메시지를 Discord Channel 로 알림.
5. **Action 3 (불합격데이터 기록):** 80점 이하이면  Processed_Results 탭에 내용을 입력합니다.
6. **Action4 (DIscord 알림)** 불합격 메시지를 Discord Channel 로 알림.
7. **보너스 설계 (실패 알림):** 시트 적재 실패 시 임시(Fallback) 시트에 데이터를 적재하거나, "저장 실패" 알림을 메신저로 보내는 대체 경로를 설계합니다.
