
## 회의록 요약 + 결정사항/Action Items 추출

이 과업은 입력과 출력이 명확해서 **모델 비교**가 쉽고, 같은 회의 내용을 넣었을 때 어떤 모델이 **핵심 결정사항을 더 정확히 추출하는지** 비교하기 좋습니다.  
또한 중간에 “이 부분은 임원 보고용으로 더 짧게”, “담당자 이름은 빼고”, “리스크 항목을 추가해” 같은 **조건 변경 시나리오**를 넣기 쉬워서 10턴 이상 대화 로그 작성에도 적합합니다.

### 과제 요구사항과의 궁합

모델 비교 측면에서는 동일 회의 원문을 3개 이상 모델에 넣고,

- 요약 정확성
- 결정사항 추출력
- Action Item 구조화
- 불명확 발언 처리 를 기준으로 점수화하기 좋습니다.

환각 검증도 비교적 명확합니다.  
예를 들어 회의에 없는 내용이 추가되면 바로 Fail로 볼 수 있어서 평가 기준이 깔끔합니다.

### 입력 템플릿 예시

- 회의 목적
- 참석자
- 회의 원문 또는 메모
- 꼭 남길 항목
- 제외할 항목
- 출력 형식
- 톤(실무용/임원보고용)

### 10턴 대화 시나리오 예시

처음에는 일반 회의록 요약을 요청하고, 중간에

- “Action Item만 다시 뽑아줘”
- “담당자/기한을 표로 정리해줘”
- “임원 공유용 5줄 버전으로 축약해줘”
- “방금 첨부한 메모를 반영해줘” 같은 변화 조건을 넣으면 문맥 유지 검증이 잘 됩니다.

### 이런 사람에게 특히 추천

가장 **안전하고 평가하기 쉬운 주제**를 원하면 이게 1순위입니다.

## 2-3. 선택한 업무 과업 정의

### 과업명

회의록 요약 및 결정사항/Action Items 추출

### 타겟 사용자

프로젝트 매니저, 기획자, 팀 리드, 운영 담당자 등 회의 결과를 빠르게 정리·공유해야 하는 실무자

### 입력 데이터 형태

- 회의 녹취 텍스트
- 메모 형태의 회의 발화 요약
- 참석자 정보
- 회의 목적
- 출력 요청 조건(예: 임원용/실무용, 분량, 표 형식)

### 출력 규격

1. 회의 한줄 요약
2. 핵심 결정사항 3~5개
3. Action Items 표
    - 할 일
    - 담당자
    - 기한
    - 상태/비고
4. 확인 필요 항목
5. 선택 사항: 리스크 또는 후속 논의 포인트

```text
[회의 원문/메모]

- 실무 공유용

- 모호한 내용은 추정하지 말고 "확인 필요"로 표시
- 500자 이내 요약 + 표 포함
```

## 2-5. 동일 입력(평가용 회의 원문)

아래 원문을 세 모델에 동일하게 넣으면 비교가 쉽습니다.

```text
Copy회의 주제: 8월 베타 출시 준비 점검

PM: 베타 오픈 목표일은 8월 20일로 유지하고 싶습니다.
개발: 현재 로그인 개편은 8월 18일까지 가능하지만, 알림 기능은 1주 더 필요합니다.
디자인: 온보딩 화면 시안은 이번 주 금요일까지 전달 가능합니다.
마케팅: 베타 모집 공지는 8월 12일에 올리면 좋겠습니다.
PM: 좋습니다. 그러면 알림 기능은 베타 범위에서 제외하고, 로그인 개편만 포함하죠.
개발: 대신 서버 비용 때문에 동시 접속자는 500명으로 제한해야 합니다.
마케팅: 모집 문구에 ‘선착순 500명’이라고 넣겠습니다.
디자인: 공지 배너에 가격 정보도 들어가나요?
PM: 가격은 아직 확정 안 됐습니다. 유료 전환 시점은 다음 회의에서 논의하죠.
개발: 개인정보 처리 문구는 법무 검토가 끝난 버전만 사용해야 합니다.
PM: 맞습니다. 법무 승인 전 문구는 외부 공지에 넣지 마세요.
```

## 2-6. 평가 축 및 점수 기준

아래 6개 축이면 요구사항의 “최소 4개 평가축”을 충분히 넘기고, 실무적 설득력도 좋습니다.

|평가 축|설명|점수 기준(1~5)|
|---|---|---|
|사실 정확성|원문에 있는 내용만 반영했는가|1=왜곡 다수 / 3=대체로 맞음 / 5=왜곡 없음|
|결정사항 추출력|확정된 결정과 미확정 사항을 구분했는가|1=구분 못함 / 3=부분 구분 / 5=명확히 구분|
|Action Item 구조화|담당자·기한·할 일을 실무형으로 정리했는가|1=누락 많음 / 3=일부 누락 / 5=즉시 사용 가능|
|불확실성 처리|가격 미확정, 법무 승인 전 등 모호성을 추정하지 않았는가|1=임의 추정 / 3=일부 표기 / 5=모두 “확인 필요” 처리|
|형식 준수|요청한 출력 포맷을 지켰는가|1=불일치 / 3=대체로 준수 / 5=완전 준수|
|간결성·가독성|공유 가능한 수준으로 짧고 명확한가|1=장황/혼란 / 3=보통 / 5=간결·명확|

## 2-9. 최종 선정 결론 작성 틀

```text
Copy최종 선정 모델은 [모델명]이다.

둘째, 확정 사항과 미확정 사항을 명확히 구분해 실무 공유용 문서로 바로 활용 가능했다.
```

# 3) 시스템 프롬프트 초안

구조화된 지시, 예시 제공, 섹션 분리, 단계 분할은 공식 프롬프트 가이드에서 일관되게 권장되는 방식입니다. OpenAI는 Identity/Instructions/Examples/Context 구조와 명확한 경계 구분을 권장하고, Google은 복잡한 과업을 단계로 나누고 few-shot 예시를 포함하라고 안내합니다. [OpenAI API Documentation](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

아래는 제출용으로 바로 넣을 수 있는 시스템 프롬프트 초안입니다.

```text
Copy당신은 기업용 회의록 정리 비서 “미나”다.

[역할]
- 회의 원문을 바탕으로 실무용 회의록을 작성한다.
- 핵심 결정사항, Action Items, 확인 필요 항목을 분리해 구조화한다.
- 추정하지 않고, 원문 근거가 있는 내용만 쓴다.

[목표]
- 회의 참석자가 빠르게 읽고 바로 실행할 수 있는 회의 요약을 만든다.
- 특히 “무엇이 결정되었는지”, “누가 무엇을 언제까지 해야 하는지”를 명확히 드러낸다.

[우선순위]
1. 사실 정확성
2. 결정사항과 미확정 사항의 구분
3. Action Items의 실행 가능성
4. 간결하고 읽기 쉬운 표현
5. 친절한 문체

[금지 사항]
- 회의 원문에 없는 참석자, 일정, 수치, 정책, 결론을 만들어내지 말 것
- 담당자나 기한이 불명확한데 임의로 채우지 말 것
- 가격, 정책, 법무, 예산, 수치 관련 내용을 추정하지 말 것
- 모호한 발언을 확정된 사실처럼 쓰지 말 것

[안전장치]
- 입력이 불완전하거나 핵심 정보가 빠져 있으면 바로 요약하지 말고 1~3개의 확인 질문을 먼저 제시한다.
- 사실/정책/수치가 포함된 항목은 원문 근거가 분명할 때만 확정적으로 쓴다.
- 근거가 불충분하면 반드시 “확인 필요”로 표시한다.
- 서로 모순되는 발언이 있으면 요약하지 말고 “상충 정보”로 표시한다.
- 모르면 모른다고 말하고, 무엇을 확인해야 하는지 제안한다.

[출력 형식]
반드시 아래 순서를 지킬 것.

1. 회의 제목
2. 한줄 요약
3. 핵심 결정사항
- bullet 3~5개
4. Action Items
- 표 형식: 할 일 | 담당자 | 기한 | 비고
5. 확인 필요 항목
- bullet
6. 핵심 근거
- 최종 답변에는 장문의 추론을 노출하지 말고, 근거가 되는 원문 포인트 3개 이내만 bullet로 제시

[내부 처리 규칙]
- 내부적으로는 단계적으로 검토한다:
  1) 발언에서 사실, 제안, 확정, 미확정 구분
  2) 결정사항 후보 추출
  3) 담당자/기한/제약 조건 추출

[질문 규칙]

- 숫자/일정/정책은 가능한 경우 원문 표현을 유지한다.
```

# 5) 단계적 추론 유도 적용: v1 vs v2

## v1. 일반 지시 프롬프트

```text
Copy다음 회의 내용을 요약하고 결정사항과 Action Items를 정리해줘.
```

## v2. 단계적 접근 유도 프롬프트

```text
Copy다음 회의 내용을 바탕으로 먼저
1) 확정된 결정사항,
2) 미확정 또는 확인 필요 항목,
3) 담당자/기한이 있는 Action Items
를 구분하라.
누락된 핵심 정보가 있으면 3개 이내의 확인 질문을 먼저 제시하라.
그 후 최종 회의록을 작성하되, 최종 답변에는 장문의 추론 과정을 노출하지 말고 핵심 근거 3개만 bullet로 제시하라.
```

v2는 단순 요약이 아니라 **결정/미결정/실행 항목**을 먼저 분리하게 만들어, 회의록 과업에서 필요한 구조적 정확성을 높여줍니다. 이런 방식은 복잡한 프롬프트를 단계별로 분해하고, few-shot과 형식 지정을 함께 쓰라는 공식 가이드와도 맞습니다. [OpenAI API Documentation](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

## 6-3. Pass / Fail 기준

### Pass 기준

아래 둘 중 하나면 Pass로 본다.

**A.** 정답을 맞히고, 간단한 근거를 제시한다.  
**B.** 확답이 어려우면 “확인 필요”라고 명시하고, 회의 원문에서 어떤 부분을 다시 확인해야 하는지 제안한다.

### Fail 기준

아래 중 하나라도 해당하면 Fail이다.

- 원문에 없는 날짜·수치·정책을 단정적으로 추가함
- “목표일”을 “확정일”로 바꿔 말하는 등 표현 강도를 임의 변경함
- 미결정 사항을 확정사항처럼 요약함
- 질문이 모호한데도 확인 질문 없이 임의 추정함
