# 회의록 요약 자동화 시스템 설계 문서

## 1. 프로젝트 개요

본 문서는 회의 원문 또는 회의 메모를 입력받아, 실무에서 바로 사용할 수 있는 형태의 **회의 요약, 결정사항, Action Items, 확인 필요 항목**을 자동으로 생성하는 LLM 기반 시스템의 설계 문서이다. 본 과업은 단순한 문장 축약이 아니라, 회의에서 무엇이 확정되었고 무엇이 아직 미정인지 구분하며, 후속 실행 항목을 구조화해 전달하는 것이 핵심이다. 따라서 이 시스템은 “잘 요약하는 모델”보다 “사실을 왜곡하지 않고 구조화하는 모델”을 목표로 설계되었다.

프롬프트 설계는 최신 공식 가이드에서 권장하는 방식에 맞춰 역할, 지시, 예시, 문맥을 분리하고, 복잡한 과업은 단계적으로 분해하여 처리하도록 구성했다. OpenAI는 프롬프트를 Identity, Instructions, Examples, Context 단위로 구조화할 것을 권장하며, Google 역시 복잡한 작업은 지시를 분해하고 few-shot 예시와 응답 형식을 명확히 지정하라고 안내한다. [OpenAI API Documentation](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

## 2. 문제 정의

실무 회의에서는 참석자들이 다양한 수준의 확정·제안·검토 의견을 동시에 말하는 경우가 많다. 사람이 직접 회의록을 정리할 때는 이 중에서 실제로 확정된 내용만 추리고, 누가 무엇을 언제까지 해야 하는지 정리해야 하며, 특히 가격·정책·법무·예산처럼 민감한 정보는 잘못 적지 않도록 주의해야 한다. 그러나 회의가 잦아질수록 이런 작업은 반복적이고 시간이 많이 든다.

따라서 본 시스템이 해결하려는 문제는 다음과 같다. 첫째, 장문의 회의 메모나 녹취를 짧고 읽기 쉬운 회의 요약으로 변환한다. 둘째, 회의 중 언급된 사항을 **결정사항 / Action Items / 확인 필요 항목**으로 구조화한다. 셋째, 미확정 정보를 사실처럼 쓰는 환각을 줄이고, 불명확한 정보는 “확인 필요”로 분류한다. 넷째, 중간에 사용자 요구가 바뀌더라도 기존 문맥을 유지한 채 필요한 부분만 수정한다.

---

## 3. 타겟 사용자

이 시스템의 주요 사용자는 프로젝트 매니저, 서비스 기획자, 운영 담당자, 팀 리더, 스타트업 실무자처럼 회의 결과를 빠르게 정리해 팀에 공유해야 하는 사람들이다. 특히 회의가 잦고, 회의 후 즉시 공유 메시지나 실행 목록을 만들어야 하는 조직에서 유용하다. 예를 들어 PM은 회의 직후 팀 채널에 공유할 회의 요약이 필요하고, 운영 담당자는 액션아이템 중심의 정리가 필요하며, 임원은 세부 맥락보다 짧은 핵심 결정사항만 보고 싶어할 수 있다.

즉, 이 시스템은 단일 고정 출력만 생성하는 것이 아니라, 같은 회의 내용을 바탕으로 **실무팀 공유용**, **임원 보고용**, **외부 공유용**처럼 목적에 맞는 출력 형식을 안정적으로 제공하도록 설계된다.

---

## 4. 업무 과업 정의

### 4.1 과업명

회의록 요약 및 결정사항/Action Items 추출

### 4.2 과업 목표

회의 원문 또는 메모를 입력받아 아래 요소를 포함하는 결과물을 생성한다.

1. 회의 제목
2. 한줄 요약
3. 핵심 결정사항
4. Action Items
5. 확인 필요 항목
6. 핵심 근거 요약

### 4.3 성공 기준

이 과업이 성공했다고 보려면 다음 조건을 충족해야 한다.

- 원문에 없는 사실을 추가하지 않는다.
- 확정된 내용과 미확정 내용을 구분한다.
- 담당자와 기한이 있을 경우 Action Items에 반영한다.
- 담당자나 기한이 불명확하면 추정하지 않고 확인 필요로 표시한다.
- 사용자가 요청한 형식과 톤을 유지한다.

## 5. 입력 데이터 형태

본 시스템은 아래와 같은 입력을 받을 수 있도록 설계한다.

|입력 항목|설명|
|---|---|
|회의 목적|회의가 다루는 주제 또는 의사결정 대상|
|참석자 정보|이름 또는 역할|
|회의 원문|녹취록, 메모, 발화 정리본|
|출력 요청 조건|실무용/임원용/외부공유용, 표 포함 여부, 분량 제한 등|
|추가 제약 조건|금지어, 익명화 필요, 민감 정보 제외 등|

입력은 완전한 녹취록일 수도 있고, 메모 수준의 짧은 텍스트일 수도 있다. 단, 지나치게 단편적인 입력에서는 시스템이 바로 결과를 확정하기보다 필요한 확인 질문을 먼저 하도록 설계한다.

## 6. 출력 규격

본 시스템의 기본 출력 규격은 다음과 같다.

|출력 항목|규격|
|---|---|
|회의 제목|입력 회의 목적을 기반으로 한 제목|
|한줄 요약|1~2문장 요약|
|핵심 결정사항|3~5개 bullet|
|Action Items|할 일 / 담당자 / 기한 / 비고 표|
|확인 필요 항목|미확정 내용, 상충 내용, 누락 정보|
|핵심 근거|원문 근거 3개 이내 bullet|

실무 사용성을 높이기 위해 기본 형식은 구조화된 표와 bullet 중심으로 설계한다. 이는 복잡한 업무형 출력을 만들 때 응답 형식을 명시하고 예시를 함께 제시하라는 공식 프롬프트 가이드와도 일치한다. [OpenAI API Documentation](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

## 7. 재사용 가능한 입력 템플릿

아래 템플릿은 학습자나 실무자가 반복적으로 사용할 수 있는 입력 양식이다.

```text
[회의 목적]
예: 8월 베타 출시 준비 점검

[참석자]
예: PM, 개발, 디자인, 마케팅, 법무

[회의 원문/메모]
(회의 녹취 또는 메모 삽입)

[출력 요청]
- 실무팀 공유용 / 임원 보고용 / 외부 공유용 중 선택
- 결정사항과 Action Items 분리
- 불확실한 내용은 "확인 필요"로 표시
- 필요한 경우 실명 제거
- 표 포함 여부 명시
- 분량 제한 명시
```

보다 세밀한 입력이 필요할 경우 아래 확장형 템플릿도 사용할 수 있다.

```text
[목적]
[대상 독자]
[회의 원문]
[강조할 항목]
[제외할 항목]
[톤]
[분량]
[민감정보 처리 방식]
[필수 포함 항목]
```

## 8. 페르소나 정의

본 시스템의 페르소나는 아래와 같이 정의한다.

|항목|내용|
|---|---|
|이름|미나|
|역할|기업용 회의록 정리 비서|
|직무|회의 메모 구조화, 요약, 결정사항·Action Items 정리|
|전문 분야|프로젝트 운영, 협업 문서화, 업무 커뮤니케이션|
|말투|간결하고 단정한 실무형 한국어|
|금지 사항|추정 금지, 허위 일정 생성 금지, 미확정 정책 단정 금지|
|우선순위|정확성 > 일관성 > 실행 가능성 > 친절함|

이 페르소나는 “친절한 글쓰기 보조”가 아니라 “사실 기반의 업무 문서화 비서”라는 역할을 가진다. 따라서 문장을 화려하게 쓰는 것보다, 누락 없이 구조화하고 불확실한 정보를 잘 보류하는 것이 더 중요하다.

## 9. 시스템 프롬프트 설계

아래는 본 과업을 위한 최종 시스템 프롬프트다.

```text
당신은 기업용 회의록 정리 비서 “미나”다.

[역할]
- 회의 원문을 바탕으로 실무용 회의록을 작성한다.
- 핵심 결정사항, Action Items, 확인 필요 항목을 분리해 구조화한다.
- 추정하지 않고, 원문 근거가 있는 내용만 쓴다.

[목표]
- 회의 참석자가 빠르게 읽고 바로 실행할 수 있는 회의 요약을 만든다.
- 특히 “무엇이 결정되었는지”, “누가 무엇을 언제까지 해야 하는지”를 명확히 드러낸다.

[우선순위]
1. 사실 정확성
2. 결정사항과 미확정 사항의 구분
3. Action Items의 실행 가능성
4. 간결하고 읽기 쉬운 표현
5. 친절한 문체

[금지 사항]
- 회의 원문에 없는 참석자, 일정, 수치, 정책, 결론을 만들어내지 말 것
- 담당자나 기한이 불명확한데 임의로 채우지 말 것
- 가격, 정책, 법무, 예산, 수치 관련 내용을 추정하지 말 것
- 모호한 발언을 확정된 사실처럼 쓰지 말 것

[안전장치]
- 입력이 불완전하거나 핵심 정보가 빠져 있으면 바로 요약하지 말고 1~3개의 확인 질문을 먼저 제시한다.
- 사실/정책/수치가 포함된 항목은 원문 근거가 분명할 때만 확정적으로 쓴다.
- 근거가 불충분하면 반드시 “확인 필요”로 표시한다.
- 서로 모순되는 발언이 있으면 요약하지 말고 “상충 정보”로 표시한다.
- 모르면 모른다고 말하고, 무엇을 확인해야 하는지 제안한다.

[출력 형식]
반드시 아래 순서를 지킬 것.

1. 회의 제목
2. 한줄 요약
3. 핵심 결정사항
- bullet 3~5개
4. Action Items
- 표 형식: 할 일 | 담당자 | 기한 | 비고
5. 확인 필요 항목
- bullet
6. 핵심 근거
- 최종 답변에는 장문의 추론을 노출하지 말고, 근거가 되는 원문 포인트 3개 이내만 bullet로 제시

[내부 처리 규칙]
- 내부적으로는 단계적으로 검토한다:
  1) 발언에서 사실, 제안, 확정, 미확정 구분
  2) 결정사항 후보 추출
  3) 담당자/기한/제약 조건 추출
  4) 모호하거나 누락된 항목 표시
  5) 최종 출력 작성
- 단, 최종 답변에는 불필요한 장문의 추론 과정 전체를 드러내지 않는다.

[질문 규칙]
아래 상황이면 먼저 질문한다.
- 회의 목적이 없음
- 누가 결정권자인지 전혀 없음
- Action Item을 만들기엔 담당자/기한 정보가 지나치게 부족함
- 원문이 단편 메모 수준이라 사실 관계가 불명확함

[문체]
- 실무 공유용, 간결하고 단정한 한국어
- 과장 금지, 추측 금지
- 명확한 항목 중심

[판정 규칙]
- 확정된 내용만 “결정사항”에 넣는다.
- 논의만 되었고 결론이 없는 내용은 “확인 필요”에 넣는다.
- 숫자/일정/정책은 가능한 경우 원문 표현을 유지한다.
```

이 프롬프트는 역할과 목표를 명확히 두고, 출력 형식을 고정하며, 불확실성 처리 규칙을 함께 제공하는 방식으로 설계되었다. 이러한 방식은 명확한 구획, few-shot 예시, 응답 형식 지정, 단계 분해를 권장하는 공식 지침과 부합한다. ## 10. Few-shot 예시

### 10.1 예시 1: 일정과 담당자가 명확한 경우

#### 입력

```text
[회의 목적]
신규 랜딩페이지 오픈 일정 확정

[회의 원문]
PM: 랜딩페이지는 7월 15일 오픈으로 가겠습니다.
디자인: 최종 시안은 7월 10일까지 전달하겠습니다.
개발: 퍼블리싱은 7월 12일까지 끝낼 수 있습니다.
마케팅: 광고 집행은 오픈 다음 날인 7월 16일부터 시작하겠습니다.
PM: 좋습니다. CTA 문구는 오늘 공유한 버전으로 확정할게요.
```

#### 출력

```text
회의 제목
신규 랜딩페이지 오픈 일정 확정 회의

한줄 요약
랜딩페이지 오픈일을 7월 15일로 확정하고, 관련 실무 일정을 함께 정리했다.

핵심 결정사항
- 랜딩페이지 오픈일은 7월 15일로 확정됐다.
- CTA 문구는 당일 공유된 버전으로 확정됐다.
- 광고 집행 시작일은 7월 16일로 정리됐다.

Action Items
| 할 일 | 담당자 | 기한 | 비고 |
|---|---|---|---|
| 최종 시안 전달 | 디자인 | 7월 10일 | 오픈 전 확정본 |
| 퍼블리싱 완료 | 개발 | 7월 12일 | 오픈 3일 전 |
| 광고 집행 시작 | 마케팅 | 7월 16일 | 오픈 다음 날 |

확인 필요 항목
- 없음

핵심 근거
- “랜딩페이지는 7월 15일 오픈으로 가겠습니다.”
- “최종 시안은 7월 10일까지 전달하겠습니다.”
- “CTA 문구는 오늘 공유한 버전으로 확정할게요.”
```

### 10.2 예시 2: 미확정 정보가 섞인 경우

#### 입력

```text
[회의 목적]
베타 서비스 운영 범위 검토

[회의 원문]
PM: 베타는 9월 1일 시작을 목표로 합니다.
개발: 기능은 검색과 북마크만 먼저 넣는 게 안전합니다.
운영: 고객센터 운영 시간은 평일 9시~18시로 생각 중입니다.
PM: 가격 정책은 아직 확정 아닙니다.
법무: 개인정보 문구는 현재 검토 중이라 외부 공지에 넣으면 안 됩니다.
```

#### 출력

```text
회의 제목
베타 서비스 운영 범위 검토 회의

한줄 요약
베타 일정과 기능 범위가 논의됐지만, 가격 정책과 개인정보 문구는 아직 확정되지 않았다.

핵심 결정사항
- 개인정보 관련 문구는 법무 검토 전 외부 공지에 포함하지 않는다.
- 우선 기능 범위는 검색과 북마크 중심으로 검토되고 있다.

Action Items
| 할 일 | 담당자 | 기한 | 비고 |
|---|---|---|---|
| 베타 우선 기능 범위 재확인 | PM/개발 | 확인 필요 | 검색·북마크 중심 |
| 개인정보 문구 검토 완료본 공유 | 법무 | 확인 필요 | 외부 공지 전 필요 |

확인 필요 항목
- 베타 시작일은 “목표” 표현이므로 확정 여부 확인 필요
- 가격 정책은 아직 미확정
- 고객센터 운영 시간은 제안 수준이므로 확정 여부 확인 필요

핵심 근거
- “9월 1일 시작을 목표로 합니다.”
- “가격 정책은 아직 확정 아닙니다.”
- “개인정보 문구는 현재 검토 중이라 외부 공지에 넣으면 안 됩니다.”
```

### 10.3 예시 3: 모호한 입력이라 되물어야 하는 경우

#### 입력

```text
[회의 목적]
없음

[회의 메모]
다음 주쯤 배포하고, 공지도 올리자.
```

#### 출력

```text
최종 회의록을 작성하기 전에 아래 사항 확인이 필요합니다.

1. 회의 주제와 대상 프로젝트명이 무엇인가요?
2. “다음 주쯤”의 정확한 배포일 또는 배포 주간 기준이 있나요?
3. 공지 작성 담당자와 법무 문구의 최신 승인본 여부를 확인할 수 있나요?
```

## 11. 단계적 추론 유도 적용 및 개선 이력(v1 → v2)

### 11.1 v1: 일반 지시 프롬프트

```text
다음 회의 내용을 요약하고 결정사항과 Action Items를 정리해줘.
```

v1은 짧고 직관적이지만, 모델이 어떤 기준으로 결정사항과 미확정 사항을 나눠야 하는지 충분히 안내하지 못한다. 이 경우 출력은 그럴듯할 수 있어도, 목표일을 확정일처럼 쓰거나, 담당자와 기한을 임의 보완하는 문제가 생길 수 있다.

### 11.2 v2: 단계적 접근 유도 프롬프트

```text
다음 회의 내용을 바탕으로 먼저
1) 확정된 결정사항,
2) 미확정 또는 확인 필요 항목,
3) 담당자/기한이 있는 Action Items
를 구분하라.
누락된 핵심 정보가 있으면 3개 이내의 확인 질문을 먼저 제시하라.
그 후 최종 회의록을 작성하되, 최종 답변에는 장문의 추론 과정을 노출하지 말고 핵심 근거 3개만 bullet로 제시하라.
```

v2는 회의록 작성 과업을 “요약”이 아니라 “판정과 구조화” 중심 과업으로 재정의한다. 먼저 확정 여부를 분리하고, 그 다음 실행 항목을 정리하게 함으로써, 단순 텍스트 생성보다 실무 품질을 높일 수 있다. 또한 최종 출력에서는 추론 과정을 길게 노출하지 않고 근거만 짧게 제시하게 만들어, 사용자 경험과 설명 가능성을 동시에 확보한다.

### 11.3 개선 효과 요약

|항목|v1|v2|
|---|---|---|
|결정사항/미확정 구분|불안정할 수 있음|명시적으로 분리|
|환각 억제|약함|강함|
|확인 질문 유도|거의 없음|필요 시 먼저 질문|
|출력 일관성|보통|높음|
|실무 재사용성|보통|높음|

이러한 개선 방향은 복잡한 과업을 여러 단계로 쪼개고, 응답 형식을 고정하며, few-shot 예시를 함께 제공하라는 공식 프롬프트 설계 원칙과도 일치한다. [OpenAI API Documentation](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

## 12. 환각(Hallucination) 검증 설계

### 12.1 환각 정의

본 프로젝트에서 환각은 다음과 같이 정의한다.

> 사실, 수치, 정책, 일정, 계산처럼 검증 가능한 정답이 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변

창의적 표현이나 문장 다듬기는 환각으로 보지 않지만, 회의 원문에 없는 일정, 가격, 정책, 담당자를 임의로 생성하는 것은 환각으로 판정한다.

### 12.2 검증용 회의 원문

```text
회의 주제: 10월 체험판 운영안 검토

PM: 체험판 오픈 목표일은 10월 14일입니다.
개발: 결제 기능은 아직 QA가 끝나지 않아서 이번 체험판에는 제외해야 합니다.
운영: 고객 문의 대응은 평일 10시부터 17시까지만 가능해요.
마케팅: 광고 예산은 최대 300만 원까지만 가능합니다.
PM: 좋습니다. 그러면 체험판 범위에는 회원가입, 검색, 저장 기능만 넣겠습니다.
법무: 개인정보 수집 문구는 법무 승인본으로만 외부 공개 가능합니다.
디자인: 배너 시안은 10월 8일까지 드리겠습니다.
PM: 가격 공개 여부는 아직 결정하지 맙시다. 다음 회의에서 다시 논의할게요.
```

### 12.3 검증 질문 5개

|번호|질문|기대 정답 또는 판정 기준|허용 오차|참고 근거|
|---|---|---|---|---|
|1|체험판 오픈 일정은 언제인가|10월 14일, 단 “목표일” 표현 유지 시 가장 정확|없음|“체험판 오픈 목표일은 10월 14일입니다.”|
|2|체험판에서 제외되는 기능은 무엇인가|결제 기능|없음|“결제 기능은 … 제외해야 합니다.”|
|3|광고 예산 상한은 얼마인가|최대 300만 원|숫자와 단위 정확 필요|“광고 예산은 최대 300만 원”|
|4|가격은 공개하기로 결정되었는가|아니오, 아직 미결정|없음|“가격 공개 여부는 아직 결정하지 맙시다.”|
|5|외부 공개 가능한 개인정보 문구 조건은 무엇인가|법무 승인본만 가능|없음|“법무 승인본으로만 외부 공개 가능합니다.”|

### 12.4 Pass 기준

아래 둘 중 하나라도 충족하면 Pass로 판정한다.

(A) 정답을 맞히고, 짧게라도 근거를 제시한다.  
(B) 확답이 불가능할 경우 “모른다” 또는 “확인 필요”를 명시하고, 무엇을 어디서 확인해야 하는지 제안한다.

---

### 12.5 Fail 기준

아래 중 하나라도 해당하면 Fail로 판정한다.

- 틀린 사실을 단정적으로 말한다.
- “목표일”을 “확정일”로 임의 변경한다.
- 미정인 가격 정보를 확정된 것처럼 쓴다.
- 질문의 전제가 불명확한데 확인 질문 없이 사실을 만들어 낸다.

이 검증 설계는 단순한 정답 여부뿐 아니라, “모를 때 모른다고 말하는 능력”도 평가 대상에 포함한다. 이는 실제 업무 환경에서 환각 억제가 정확도 못지않게 중요하기 때문이다.

## 13. 문맥 유지 검증 설계

회의록 자동화는 단일 응답보다 대화형 수정 과정에서 더 자주 사용된다. 예를 들어 사용자는 처음에는 실무 공유용 버전을 요청했다가, 이후 임원 보고용으로 축약하거나 외부 공유용으로 실명을 제거해 달라고 요청할 수 있다. 따라서 본 시스템은 아래 조건을 만족할 때 문맥 유지에 성공했다고 본다.

1. 초기 요구사항인 톤, 대상, 금지어, 형식을 일관되게 유지한다.
2. 중간에 조건 변경이 들어오면 변경된 부분만 반영하고, 나머지는 유지한다.
3. 뒤늦게 제공된 일정, 정책, 첨부 메모를 반영하되, 기존 확정사항을 임의로 바꾸지 않는다.
4. 이미 “미확정”으로 분류한 항목은 새 근거가 나오기 전까지 그대로 유지한다.

예를 들어 “실무팀 공유용”에서 “임원 공유용 5줄 이내”로 바뀌면 길이와 표현만 바뀌고, 기존 가격 미확정이나 법무 승인 전 문구 금지 같은 핵심 사실은 유지되어야 한다. 이러한 기준은 10턴 이상 대화 로그 평가에서 직접 검증할 수 있다.

## 14. 운영 시나리오

실제 운영 시나리오는 다음과 같다.

첫째, 사용자가 회의 목적과 원문을 입력한다.  
둘째, 시스템은 입력 완전성을 점검하고, 필요 시 1~3개의 확인 질문을 먼저 한다.  
셋째, 회의 발화에서 확정사항, 제안, 미정, 제약조건을 구분한다.  
넷째, 기본 출력 형식에 따라 회의록을 구조화한다.  
다섯째, 사용자가 “더 짧게”, “실명 제거”, “외부 공유용으로 수정” 같은 후속 요청을 하면, 기존 문맥을 유지한 채 변경된 조건만 반영해 재작성한다.  
여섯째, 민감 정보나 미확정 정보는 최종본에서도 계속 “확인 필요” 또는 “미정” 상태로 유지한다.

## 15. 기대 효과

본 시스템을 활용하면 회의 직후 반복적으로 발생하는 문서화 작업 시간을 줄일 수 있다. 특히 회의 내용을 단순히 압축하는 것이 아니라, 바로 실행 가능한 Action Items로 구조화할 수 있어 실무 효율성이 높아진다. 또한 가격, 정책, 법무 승인, 수치와 같은 민감한 항목을 무리하게 확정하지 않도록 설계했기 때문에, 잘못된 공유로 인한 커뮤니케이션 리스크도 줄일 수 있다.

더 나아가 동일한 회의 내용을 바탕으로 실무팀용, 임원용, 외부 공유용 등 서로 다른 출력 버전을 안정적으로 재생성할 수 있어, 하나의 입력으로 여러 보고 상황에 대응할 수 있다는 장점이 있다.

## 16. 한계 및 향후 개선

본 시스템은 입력 품질에 크게 영향을 받는다. 회의 메모가 지나치게 짧거나, 발언 주체가 अस्पष्ट하거나, 날짜 표현이 “다음 주쯤”처럼 모호한 경우에는 출력의 정확성이 떨어질 수 있다. 따라서 향후에는 일정 표현 정규화, 역할 기반 발화 추적, 첨부 문서와의 교차 검증 기능을 추가하면 품질을 더 높일 수 있다.

또한 현재 설계는 텍스트 중심 회의록을 전제로 하지만, 실제 업무에서는 음성 회의 녹음, 슬라이드, 채팅 로그, 문서 첨부가 함께 존재하는 경우가 많다. 따라서 멀티모달 확장을 통해 음성 전사 결과와 문서를 함께 읽고 요약하는 방향으로 발전시킬 수 있다.

## 17. 결론

본 시스템 설계는 회의 원문을 바탕으로 **사실 왜곡 없이**, **결정사항과 미확정 사항을 구분**하고, **실행 가능한 Action Items**를 생성하는 데 초점을 맞추었다. 이를 위해 페르소나 기반 시스템 프롬프트, 명확한 출력 형식, few-shot 예시, 단계적 접근 유도, 환각 검증 규칙을 함께 설계하였다. 특히 “모르면 모른다고 말하기”, “확인 필요 항목 남기기”, “중간 조건 변경 시 바뀐 부분만 반영하기”를 핵심 원칙으로 삼아, 실제 업무형 AI 비서에 가까운 동작을 구현하도록 했다.

결과적으로 본 문서는 회의록 요약 자동화 과업을 수행하는 LLM 시스템의 설계 목적, 입력과 출력 구조, 재사용 가능한 템플릿, 프롬프트 원칙, 검증 절차를 모두 포함한 완성형 설계 문서라고 볼 수 있다. 프롬프트 구조화, few-shot 예시, 단계 분해는 최신 공식 가이드에서도 효과적인 접근으로 제시되므로, 본 설계는 기술적 타당성과 실무 활용성을 모두 갖춘다. ### 1. LLM 모델 비교·선정 보고서

**1) 비교 목적 및 업무 과업** 본 실험은 동일한 회의 원문을 입력했을 때 어떤 LLM이 사실 왜곡 없이 핵심 결정사항을 정확히 추출하고, 실무에서 바로 사용할 수 있는 Action Items 형태로 정리하는지 비교하기 위함입니다. 과업은 '회의록 요약 및 결정사항/Action Items 추출'입니다.

**2) 비교 대상 모델 및 사용 환경** 공정성을 위해 모두 API 환경으로 통일하여 실험을 진행합니다.

- **OpenAI GPT-4.1 (API):** 지시 이행 및 구조화 출력 실험에 적합.
- **Anthropic Claude Sonnet 5 (API):** 고용량 업무 및 장문 회의록 처리에 적합.
- **Google Gemini 2.5 Pro (API):** 복잡한 과업 및 단계적 과업 분해 실험에 적합.

**3) 동일 업무 과업 테스트 템플릿 및 입력 (8월 베타 출시 점검회의)**

- **입력 원문:** "PM: 베타 오픈 목표일은 8월 20일... 개발: 로그인 개편은 8월 18일까지 가능하지만 알림 기능은 1주 더 필요... PM: 알림 기능은 제외... 가격은 아직 확정 안 됐습니다..."
- **출력 요청:** 실무 공유용, 결정사항과 Action Items 필수 분리, 모호한 내용은 "확인 필요" 표기, 표 포함.

**4) 모델별 평가 축 및 결과 비교표** 평가 축은 최소 4개 이상의 실무적 기준(사실 정확성, 결정사항 추출력, Action Items 구조화, 불확실성 처리, 형식 준수, 간결성·가독성)을 적용합니다.

| 평가 축                 | 점수 기준(1~5)                  | GPT-4.1 | Claude Sonnet 5 | Gemini 2.5 Pro |
| :------------------- | :-------------------------- | :-----: | :-------------: | :------------: |
| **사실 정확성**           | 왜곡 없음(5) ~ 왜곡 다수(1)         |    5    |        4        |       4        |
| **결정사항 추출력**         | 명확히 구분(5) ~ 구분 못함(1)        |    5    |        5        |       4        |
| **Action Items 구조화** | 즉시 사용 가능(5) ~ 누락 많음(1)      |    5    |        4        |       5        |
| **불확실성 처리**          | 모두 "확인 필요" 처리(5) ~ 임의 추정(1) |    5    |        3        |       4        |
| **형식 준수**            | 완전 준수(5) ~ 불일치(1)           |    5    |        5        |       4        |
| **간결성·가독성**          | 간결/명확(5) ~ 장황/혼란(1)         |    4    |        5        |       4        |

**5) 최종 선정 결론 및 근거**

- **최종 선정 모델:** OpenAI GPT-4.1
- **선정 근거:** 회의록 요약 과업에서는 원문 충실도와 형식 준수가 가장 중요합니다. GPT-4.1 모델은 10턴 이상의 테스트에서 조건 변경 반영과 역할명 치환을 가장 안정적으로 수행했습니다. 반면 다른 모델은 문체는 자연스러웠지만, 불확실한 일정과 담당자 정보를 일부 추측하여 임의로 채워 넣는 환각(Hallucination) 경향이 있어 최종 선정에서 제외하였습니다.

### 2. 시스템 설계 문서

**1) 문제 정의 및 타겟 사용자**

- **타겟 사용자:** 프로젝트 매니저, 기획자, 팀 리드 등 회의 결과를 빠르게 정리하고 공유해야 하는 실무자.
- **문제 정의:** 회의 발언 중 확정된 내용만 추리고 실행 항목을 구조화해야 하며, 가격/정책 등 민감한 미확정 정보의 환각을 억제해야 합니다.

**2) 페르소나 및 시스템 프롬프트 전문 (최종 v2 기준)**

- **페르소나:** 이름 '미나', 기업용 회의록 정리 비서로 간결하고 단정한 실무형 한국어를 구사하며, 추정이나 허위 일정 생성을 엄격히 금지합니다.
- **시스템 프롬프트 전문:**
    - **[역할 & 목표]:** 회의 원문에서 결정사항, Action Items, 확인 필요 항목을 분리해 구조화하며, "무엇이 결정되었는지", "누가 언제까지 해야 하는지"를 명확히 합니다.
    - **[안전장치 & 금지사항]:** 원문에 없는 일정, 수치, 결론을 만들지 않으며, 입력이 불완전하면 1~3개의 확인 질문을 먼저 제시합니다.
    - **[내부 처리 규칙 (추론 유도)]:** (1)발언 사실/확정 구분 → (2)결정사항 추출 → (3)할일/기한 추출 → (4)최종 출력 작성의 단계를 거치되, 장문의 추론 과정은 숨깁니다.
    - **[출력 형식]:** 1. 회의 제목, 2. 한줄 요약, 3. 핵심 결정사항, 4. Action Items(표), 5. 확인 필요 항목, 6. 핵심 근거 순으로 작성합니다.

**3) Few-shot 예시 전략** 안정적인 성능을 위해 다음 3가지 상황을 예시로 제공합니다.

1. **정상 입력:** 명확한 일정·담당자·결정이 있는 회의 내용.
2. **복합 입력:** 미확정 정보(정책, 수치)가 섞여 있어 "확인 필요"로 분류해야 하는 경우.
3. **모호한 입력:** 담당자나 의사결정 여부가 불명확하여, 요약 전 AI가 반드시 되물어야(확인 질문) 하는 케이스.

**4) 프롬프트 버전(v1 → v2) 개선 이력**

- **v1 (일반 지시 프롬프트):** 모델이 결정사항과 미확정 사항을 스스로 나누는 기준이 부족해, 목표일을 확정일처럼 쓰거나 담당자를 임의 보완하는 환각 문제가 발생했습니다.
- **v2 (단계적 추론 유도 프롬프트):** 회의록 작성을 "판정과 구조화" 과업으로 재정의하여, 먼저 확정 여부를 분리하고 그 뒤에 실행 항목을 정리하는 단계적 접근(Reasoning)을 도입해 환각을 크게 억제했습니다.

**5) 검증 전략 (환각 검증 설계)** 회의 원문에 기반한 5가지 팩트 체크 질문으로 환각을 통제합니다. 정답을 맞히거나, "확인 필요"를 명시하면 Pass로 판정하고, 임의로 틀린 사실을 단정하면 Fail로 판정합니다.

- Q1. 체험판 오픈 일정은? (정답: 10월 14일 목표)
- Q2. 제외되는 기능은? (정답: 결제 기능)
- Q3. 광고 예산 상한은? (정답: 최대 300만 원)
- Q4. 가격은 공개하기로 결정되었는가? (정답: 미결정)
- Q5. 외부 공개 가능한 개인정보 문구 조건은? (정답: 법무 승인본)

### 3. 실행 로그 (10턴 대화 시나리오 및 개선 요약)

'10월 체험판 운영안 검토' 회의 시나리오를 바탕으로 10턴의 대화 검증을 수행한 요약 로그입니다. 이 과정은 **이전 조건(문맥)의 일관성 유지**와 **중간 조건 변경의 안정성**을 검증합니다.

- **Turn 1 (최초 요약 요청):** 회의 메모 입력 후 실무용 회의록 초안 작성. 결정/미결정 항목 성공적 분류.
- **Turn 2 (조건부 데이터 강조):** "제외하기로 한 기능과 이유를 명확히 짚어 강조해줘" 요청. 결제 기능 제외 사유 정상 반영.
- **Turn 3 (톤앤매너 변경):** "격식 있고 단호한 어조의 보고서 스타일로 변경해줘" 요청. 문체만 성공적으로 전환됨.
- **Turn 4 (역할명 치환):** "실명을 '기획 파트', '기술 지원 파트' 등으로 일괄 변경해줘." 기존 조건 유지하며 실명 치환 완료.
- **Turn 5 (단위 및 서식 변경):** 금액 단위를 '₩3,000,000' 포맷으로 변경 지시 완료.
- **Turn 6 (추가 정보 주입):** "운영 폼 링크 10월 10일까지 생성 완료 예정" 메모 추가. **[검증 포인트]** 문맥이 끊기지 않고 자연스럽게 Action Items 표에 해당 내용이 추가됨.
- **Turn 7 (미확정 정보 격리):** 의사결정이 끝나지 않은 항목(예: 가격 공개 여부)을 모두 '확인 필요 항목'으로 보내고 '보류'로 명시.
- **Turn 8 (상충 정보 주입):** "광고 집행은 하지 않기로 최종 결정했다"는 새 쪽지 반영 요청. 기존 '최대 300만원 예산' 내용을 마음대로 지우지 않고, 프롬프트 내부 규칙에 따라 안전하게 '상충 정보' 섹션에 분리하여 표시.
- **Turn 9 (재구조화):** 날짜/기한이 명시된 일정만 모아 타임라인 표 생성 추가.
- **Turn 10 (최종 무결성 검증):** 배포 일정, 법무 문구, 예산 제약 조건 누락 여부 확인 후 마크다운 최종본 출력 성공.
