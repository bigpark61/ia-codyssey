
좋습니다. 이 과제는 **“프롬프트를 잘 쓰는 법”을 설명하는 과제**가 아니라, 실제로 **비교 → 설계 → 검증 → 개선**의 전 과정을 재현 가능한 방식으로 남기는 과제**입니다.**  
따라서 핵심은 멋진 답변 1개를 만드는 것이 아니라, **왜 그 답변이 나왔는지, 왜 그 모델과 프롬프트를 선택했는지, 어떻게 검증했는지**를 문서화하는 데 있습니다.

아래처럼 진행하면 가장 안정적으로 과제를 완성할 수 있습니다.

---

# 1. 과제를 한 문장으로 정리하면

이 과제는 다음 3가지를 하나의 흐름으로 묶는 작업입니다.

**첫째**, 동일한 업무 과업을 3개 이상의 LLM에 시켜서 비교하고,  
**둘째**, 그중 가장 적합한 모델을 골라 시스템 프롬프트와 Few-shot을 설계하고,  
**셋째**, 실제 10턴 이상 대화로 성능과 환각 여부를 검증한 뒤 개선 이력을 남기는 것입니다.

즉, 결과물은 3개이지만 실제 절차는 하나의 파이프라인으로 이어집니다.

**모델 비교 → 최적 모델 선정 → 프롬프트 설계 → 테스트 → 개선 → 최종 문서화**

이 접근은 공식 프롬프팅 가이드들이 공통으로 권장하는 방식과도 맞습니다. 명확한 역할 지정, 예시 제공, 출력 형식 제약, 복잡한 작업의 단계 분해가 품질과 일관성을 높인다고 안내합니다. [OpenAI API](https://developers.openai.com/api/docs/guides/prompt-engineering) [Claude Platform Docs](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

# 2. 가장 먼저 해야 할 일: 과업을 하나로 좁히기

이 과제에서 가장 흔한 실패는 **업무 과업을 너무 넓게 잡는 것**입니다.  
예를 들어 “업무 자동화 전반”처럼 잡으면 비교 기준도 흐려지고, 시스템 프롬프트도 산만해지고, 10턴 대화 검증도 애매해집니다.

그래서 먼저 아래 4개 중 **하나만** 고르는 것이 좋습니다.

| 후보 과업     | 장점                              | 난이도 | 추천도 |
| --------- | ------------------------------- | --- | --- |
| 메일 초안 작성  | 톤 조절, 대상 맞춤, 조건 변경 테스트가 쉬움      | 하   | 중   |
| 회의록 요약    | 결정사항/Action Items/리스크 추출 평가가 쉬움 | 하   | 상   |
| 보고서 개요 생성 | 구조화 능력, 논리성 평가가 쉬움              | 중   | 상   |
| 요구사항 정리   | 사용자 스토리/수용 기준/누락 질문 테스트에 좋음     | 중   | 상   |

실제 과제 완성도만 보면 **“회의록 요약”** 또는 **“요구사항 정리”**가 가장 유리합니다.  
이유는 평가 축을 만들기 쉽고, 환각 검증도 비교적 명확하기 때문입니다.

예를 들어 **회의록 요약**을 선택하면 과제 전체 구조가 아주 자연스럽게 이어집니다.

- 모델 비교: 누가 더 잘 요약하는가
- 시스템 프롬프트: PM 출신 요약 코치 페르소나
- Few-shot: 좋은 요약 예시 3개
- Reasoning Prompt: 먼저 결정/액션/리스크 구분 → 누락 질문 → 템플릿 출력
- 환각 검증: 회의 내용에 없는 내용을 지어내는지 확인
- 10턴 로그: 중간에 톤, 대상, 금지어, 추가 메모가 바뀌는 상황 테스트

# 3. 전체 수행 절차: 7단계 파이프라인

## 1단계. 문제 정의서 1장 만들기

먼저 아주 짧은 **문제 정의서**를 만듭니다.  
이건 이후의 모든 문서가 흔들리지 않게 해주는 기준점입니다.

여기에는 최소한 아래 4개를 적습니다.

- **타겟 사용자**: 누가 쓰는가
- **입력 데이터 형태**: 어떤 정보를 주는가
- **출력 규격**: 어떤 형식으로 답해야 하는가
- **성공 기준**: 무엇을 잘하면 좋은 결과인가

예시로 쓰면 이렇게 됩니다.

**과업:** 회의록 요약  
**타겟 사용자:** 스타트업 PM, 팀 리드  
**입력:** 회의 메모, 녹취 요약, 메신저 발췌  
**출력:** 결정사항 / Action Items / 리스크 / 확인 필요  
**성공 기준:** 추측 금지, 담당자·기한 분리, 문맥 유지, 형식 일관성

이 단계가 중요한 이유는 이후 모델 비교도, 프롬프트 설계도, 환각 테스트도 모두 이 정의를 기준으로 평가해야 하기 때문입니다.

---

## 2단계. 입력 템플릿을 먼저 고정하기

모델을 비교하기 전에 반드시 **입력 템플릿**을 고정해야 합니다.  
그래야 3개 이상의 모델에 **동일 과업, 동일 조건**을 적용할 수 있습니다.

예를 들어 회의록 요약 과업이라면 입력 템플릿은 이렇게 만들 수 있습니다.

```text
[업무 과업]
회의록 요약

[원문]
{회의 메모 또는 녹취 요약}

[원하는 결과]
결정사항 / Action Items / 리스크 / 후속 일정

[톤]
간결하고 사내 공유용

[금지]
추측성 표현 금지, 실명 대신 역할명 사용

[확인 질문 규칙]
정보가 부족하면 최대 3개까지 질문 후 요약 시작
```

이 템플릿을 먼저 고정하면 좋은 점이 두 가지 있습니다.

첫째, 모델 비교에서 공정성이 생깁니다.  
둘째, 나중에 시스템 프롬프트와 Few-shot을 설계할 때도 무엇을 제어해야 하는지가 선명해집니다.

---

## 3단계. 모델 비교 실험 설계

이제 3개 이상의 모델을 고릅니다.  
과제 요구상 반드시 기록해야 하는 항목은 다음입니다.

- 모델명
- 요금제(무료/유료)
- 사용 채널(웹/앱/API)
- 사용 날짜
- 주요 설정(temperature, top_p, max_tokens 등 가능 범위)
- 무료 버전 한계 1줄

여기서 중요한 건 **“최신이냐”보다 “비교가 가능하냐”**입니다.  
즉, 서로 너무 다른 환경을 섞으면 점수 해석이 어려워집니다.

예를 들면 다음 조합이 무난합니다.

- ChatGPT 계열 1개
- Claude 계열 1개
- Gemini 계열 1개

그리고 동일한 테스트 케이스를 3~5개 정도 준비합니다.

### 추천 테스트 케이스 구성

1. **기본형**: 정보가 충분한 회의록
2. **모호형**: 누락 정보가 있어 질문이 필요한 경우
3. **장문형**: 길고 중복이 많은 회의록
4. **조건변경형**: 중간에 톤/형식 수정 요청
5. **사실검증형**: 날짜, 수치, 정책성 문장이 포함된 회의 메모

---

## 4단계. 평가 축을 먼저 만든 뒤 채점하기

여기서 핵심은 “느낌”이 아니라 **평가 축**입니다.  
최소 4개 이상이 요구되지만, 실제로는 6개 정도가 가장 안정적입니다.

추천 축은 아래와 같습니다.

| 평가 축      | 설명                           |
| --------- | ---------------------------- |
| 정확성       | 원문에 없는 정보 추가 여부, 사실/수치 오류 여부 |
| 형식 준수     | 템플릿 형식을 잘 지키는가               |
| 한국어 자연스러움 | 읽기 쉬운가, 어색하지 않은가             |
| 문맥 유지     | 이전 조건을 기억하는가                 |
| 누락 감지     | 정보가 부족할 때 질문하는가              |
| 속도/안정성    | 응답이 안정적이고 지나치게 흔들리지 않는가      |

점수는 1~5점으로 주되, 반드시 **근거 한 줄**을 붙입니다.

예시:

- 정확성 4점: 결정사항은 잘 추출했지만, 후속 일정 1개를 추측해 생성함
- 형식 준수 5점: 요구한 4개 섹션을 정확히 분리
- 문맥 유지 3점: 8턴 이후 역할명 규칙이 무너짐

이렇게 해야 최종 결론이 설득력을 갖습니다.

---

## 5단계. 최종 선정 모델을 고른 뒤 시스템을 설계하기

모델 비교가 끝나면 바로 프롬프트를 쓰는 것이 아니라, 먼저 **“이 모델을 어떤 역할로 쓸지”**를 설계해야 합니다.

여기서 필요한 것이 **페르소나 정의**입니다.

예를 들어 회의록 요약 과업이라면:

- **이름:** 민준
- **역할:** 프로젝트 매니저 출신 업무 자동화 코치
- **전문 분야:** 회의록 구조화, 액션 아이템 정리, 리스크 식별
- **말투:** 간결하고 명확한 사내 보고체
- **금지 사항:** 추측 금지, 실명 노출 금지, 근거 없는 일정 생성 금지
- **우선순위:** 정확성 > 형식 준수 > 친절함

이렇게 두는 이유는, 시스템 프롬프트의 목적이 단순히 “똑똑하게 만들어라”가 아니라 **행동 일관성**을 강제하는 것이기 때문입니다. 역할 지정, 예시 제공, 구조화된 태그/구분, 명시적 규칙은 결과 품질과 형식 안정성을 높이는 데 도움이 됩니다. [OpenAI API](https://developers.openai.com/api/docs/guides/prompt-engineering) [Claude Platform Docs](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)

## 6단계. 시스템 프롬프트를 v1 → v2로 개선하기

이 과제의 핵심 중 하나가 **개선 이력**입니다.  
즉, 처음부터 완벽한 프롬프트를 쓰는 게 아니라, 왜 바꿨는지를 보여줘야 합니다.

### v1: 일반 지시형

```text
당신은 회의록 요약 도우미입니다.
아래 회의 내용을 요약해서 결정사항, 액션 아이템, 리스크를 정리하세요.
가능하면 간결하게 작성하세요.
```

이 프롬프트의 문제는 다음과 같습니다.

- 정보가 부족해도 질문 규칙이 없음
- 추측 금지 규칙이 약함
- 역할명 사용 같은 형식 통제가 없음
- 사실/수치 처리 규칙이 없음
- 최종 출력 형식이 느슨함

### v2: 단계적 접근 유도형

```text
당신은 프로젝트 매니저 출신 업무 자동화 코치다.
목표는 회의 메모를 사내 공유용 요약으로 변환하는 것이다.

우선순위:
1) 정확성
2) 형식 준수
3) 간결성

규칙:
- 원문에 없는 내용은 추측하지 말 것
- 실명 대신 역할명 사용
- 날짜/수치/정책성 문구는 원문 근거가 없으면 "확인 필요"로 표기
- 정보가 부족하면 최대 3개까지 확인 질문 후 작성
- 내부적으로는 단계적으로 검토하되, 최종 답변에는 장황한 추론을 노출하지 말 것

작업 절차:
1) 결정사항/액션 아이템/리스크/확인 필요 항목을 구분
2) 누락 정보가 있으면 확인 질문 생성
3) 충분한 정보가 있으면 템플릿에 맞춰 출력

출력 형식:
[결정사항]
[Action Items]
- 담당:
- 기한:
- 작업:

[리스크]
[확인 필요]
[핵심 근거 3개]
```

이 개선 방향은 공식 가이드들이 공통으로 말하는 “명확한 지시, 구조화, Few-shot 활용, 복잡한 작업의 단계 분해”와 정확히 맞닿아 있습니다. Google은 복잡한 작업을 단계별 프롬프트로 분리하거나 체인으로 구성하라고 권장하고, OpenAI는 예시와 명확한 출력 형식을 강조하며, Anthropic은 역할 설정과 구조화 태그, 3~5개 예시의 효과를 설명합니다. [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies) [OpenAI API](https://developers.openai.com/api/docs/guides/prompt-engineering) [Claude Platform Docs](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)

## 7단계. Few-shot 예시 3개를 붙이기

Few-shot은 이 과제에서 사실상 **성능 안정화 장치**입니다.  
좋은 예시를 넣으면 모델이 형식, 범위, 톤, 예외 처리를 더 잘 따라옵니다. OpenAI는 Few-shot이 새 작업 패턴을 익히게 도와준다고 설명하고, Google은 Few-shot 예시를 항상 포함하는 것이 효과적이라고 권장합니다. [OpenAI API](https://developers.openai.com/api/docs/guides/prompt-engineering) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

예시는 이렇게 구성하면 됩니다.

### 예시 1. 정상 입력

충분한 정보가 있는 회의 메모 → 바로 템플릿 요약

### 예시 2. 장문 입력

중복과 잡담이 많은 회의 메모 → 핵심만 압축

### 예시 3. 모호한 입력

기한, 담당자, 의사결정 여부가 불명확 → 요약하지 말고 확인 질문 먼저

특히 **예시 3**이 매우 중요합니다.  
과제 요구사항에서 “모호한 입력일 때 AI가 되물어야 하는 지점”을 보여줘야 하기 때문입니다.

# 4. 환각 검증은 어떻게 해야 하나

이 과제에서 환각 검증은 단순히 “틀리면 감점”이 아닙니다.  
정확히는 **검증 가능한 질문에서, 근거 없이 틀린 답을 확신하는지**를 보는 것입니다.

Google Cloud는 AI hallucination을 **잘못되거나 오해를 부르는 결과를 생성하는 것**으로 설명하며, 원인으로 불완전한 데이터와 grounding 부족을 듭니다. 따라서 이 과제에서는 **정답을 맞히는 것만큼이나, 모르면 모른다고 말하고 확인 절차를 제시하는 것**이 중요합니다. [cloud.google.com](https://cloud.google.com/discover/what-are-ai-hallucinations)

## 검증 질문 설계 방식

질문은 반드시 **정답 또는 판정 기준이 존재하는 것**으로 만드세요.

예를 들면:

- 회의 메모에 나온 마감일은 언제인가?
- 출장비 예외 규정은 무엇인가?
- 제안된 예산 총액은 얼마인가?
- 정책상 승인 주체는 누구인가?
- 액션 아이템 담당자는 누구인가?

각 질문마다 아래를 기록합니다.

|항목|내용|
|---|---|
|검증 질문|예: “출장비 예외 조건이 뭐야?”|
|기대 정답/판정 기준|규정 문서의 예외 조항과 일치해야 함|
|허용 오차|금액 계산이면 ±0원, 날짜는 정확 일치|
|참고 근거|규정 PDF, 회의 메모, 표준 문서|
|Pass 기준|맞히거나, 모른다고 하고 확인 절차 제시|
|Fail 기준|틀린 사실을 단정적으로 말함|

이렇게 하면 보고서가 매우 탄탄해집니다.

# 5. 10턴 이상 대화 로그는 어떻게 설계해야 하나

이 부분도 많은 분들이 실수합니다.  
그냥 아무 대화 10턴을 채우는 게 아니라, **문맥 유지 능력**을 검증하도록 설계해야 합니다.

가장 좋은 방식은 **시나리오 기반 대화**입니다.

예를 들어:

### 추천 시나리오

1. 사용자가 회의 메모를 제공
2. AI가 누락 정보 질문
3. 사용자가 담당자 정보 추가
4. AI가 1차 요약
5. 사용자가 톤을 “더 단호하게” 변경
6. AI가 톤만 수정
7. 사용자가 실명 대신 역할명 사용 요청
8. AI가 기존 내용은 유지하고 이름만 치환
9. 사용자가 뒤늦게 새 메모 1개 추가
10. AI가 기존 요약에 새 정보만 반영
11. 사용자가 정책상 확정되지 않은 사항은 “확인 필요”로 바꾸라고 요청
12. AI가 전체 형식 유지하며 수정

이렇게 해야 문맥 유지 기준을 판정할 수 있습니다.

### Pass 판정 포인트

- 처음 정한 톤을 계속 유지하는가
- 중간에 바뀐 조건만 수정하는가
- 나머지 기존 요구사항은 유지하는가
- 원문에 없는 내용을 추가하지 않는가

# 6. 최종 산출물은 어떻게 묶으면 되나

실제로 제출물은 3개지만, 아래처럼 작성하면 깔끔합니다.

---

## A. 모델 비교·선정 보고서

### 추천 목차

1. 비교 목적
2. 선택한 업무 과업
3. 테스트 입력 템플릿
4. 비교 대상 모델 및 사용 환경
5. 평가 축 정의
6. 모델별 점수표
7. 모델별 결과 요약
8. 최종 선정 결론

### 결론 작성 팁

결론은 반드시 **“왜 이 모델이 이 과업에 최적이었는가”**를 3줄 이상 쓰세요.

예:

- 회의록 요약 과업에서는 원문 충실도와 형식 준수가 가장 중요했다.
- 선정 모델은 10턴 테스트에서 조건 변경 반영과 역할명 치환을 가장 안정적으로 수행했다.
- 다른 모델은 문체는 자연스러웠지만, 일정과 담당자 정보를 일부 추측하는 경향이 있어 제외했다.

---

## B. 시스템 설계 문서

### 추천 목차

1. 문제 정의
2. 타겟 사용자
3. 입력 데이터 형태
4. 출력 규격
5. 입력 템플릿
6. 페르소나 정의
7. 시스템 프롬프트
8. Few-shot 예시 3개
9. 환각 검증 설계
10. v1 → v2 개선 이력
11. 최종 프롬프트 전문

이 문서의 핵심은 **설계 이유**를 남기는 것입니다.  
그냥 프롬프트만 적지 말고, 왜 그 규칙을 넣었는지 써야 합니다.

---

## C. 실행 로그

### 추천 목차

1. 시나리오 설명
2. 원본 로그(10턴 이상)
3. 대화 중 발생한 문제
4. 수정한 프롬프트/규칙
5. 수정 후 결과 요약
6. 문맥 유지 판정
7. 환각 여부 판정

여기서 중요한 건 **원본 로그는 남기되, 제출본은 읽기 좋게 정리**하는 것입니다.

# 7. 가장 효율적인 작업 순서

실무적으로는 아래 순서가 제일 효율적입니다.

### 1일차

과업 선택 → 입력 템플릿 작성 → 테스트 케이스 3~5개 작성

### 2일차

모델 3종 테스트 → 점수표 작성 → 최종 모델 선정

### 3일차

페르소나 정의 → 시스템 프롬프트 v1 작성 → Few-shot 3개 작성

### 4일차

v2 개선 → 환각 검증 질문 5개 실행 → 결과 기록

### 5일차

10턴 이상 대화 로그 수행 → 문제 지점 수정 → 산출물 3종 정리

# 8. 채점받기 좋은 포인트

이 과제는 단순히 “좋은 답변을 얻었다”보다 아래 요소가 강하면 훨씬 좋아집니다.

첫째, **비교의 공정성**입니다.  
동일 입력, 동일 과업, 동일 평가 축을 유지해야 합니다.

둘째, **개선의 논리성**입니다.  
왜 v1이 부족했고, v2에서 무엇을 추가했는지 분명해야 합니다.

셋째, **환각 대응 태도**입니다.  
정답을 맞히는 것만큼, 모르면 확인하겠다고 말하는 설계가 중요합니다.

넷째, **문맥 유지 검증**입니다.  
10턴 대화를 그냥 길게 하는 게 아니라, 조건 변경과 추가 정보 상황을 넣어야 합니다.

# 9. 한 번에 이해할 수 있는 요약 프레임

아주 짧게 정리하면 이 과제는 아래 프레임으로 하면 됩니다.

**1. 과업 하나 선택**  
→ 회의록 요약 / 요구사항 정리 같은 좁은 업무

**2. 동일 입력으로 모델 3종 비교**  
→ 평가 축 4개 이상, 점수와 근거 기록

**3. 최적 모델 1개 선정**  
→ 왜 이 모델이 적합한지 설명

**4. 페르소나 + 시스템 프롬프트 설계**  
→ 역할, 목표, 형식, 안전장치, 사실 처리 규칙 포함

**5. Few-shot 3개 추가**  
→ 정상/장문/모호 입력 포함

**6. v1 → v2 개선 이력 작성**  
→ 단계적 접근 유도, 추론 노출 최소화

**7. 환각 검증 5개 이상 수행**  
→ 맞히거나, 모르면 확인 절차 제시

**8. 10턴 이상 대화 로그로 문맥 유지 검증**  
→ 조건 변경, 추가 정보 반영 테스트

**9. 산출물 3종 정리**  
→ 비교 보고서 / 시스템 설계 문서 / 실행 로그

# 10. 제가 추천하는 가장 안정적인 과제 수행 전략

제 추천은 아래입니다.

**과업은 “회의록 요약”으로 선택하고,**  
**최종 목표를 “결정사항 / Action Items / 리스크 / 확인 필요” 4개 섹션 출력으로 고정**하세요.

이 방식은

- 모델 비교가 쉽고,
- Few-shot 작성이 쉽고,
- 환각 검증이 쉽고,
- 10턴 대화에서 조건 변경 테스트가 자연스럽고,
- 최종 문서도 깔끔하게 정리됩니다.

즉, **과제 요구사항을 가장 빠짐없이 충족시키기 좋은 선택**입니다.
