## 0. 문서 개요

본 제출물은 “보고서 개요 생성”을 업무 자동화 과업으로 선정하여 수행한 결과를 정리한 최종 문서이다. 문서는 과제 요구사항에 맞추어 세 부분으로 구성된다. 첫째, 3종 이상의 LLM을 동일 과업에 투입하여 비교·선정한 **모델 비교·선정 보고서**를 제시한다. 둘째, 선정된 모델을 기준으로 페르소나, 시스템 프롬프트, Few-shot 예시, 환각 검증 전략, 프롬프트 개선 이력을 정리한 **시스템 설계 문서**를 제시한다. 셋째, 실제 10턴 이상의 상호작용을 바탕으로 문맥 유지와 문제 수정 과정을 확인할 수 있는 **실행 로그**를 제시한다.

본 과제는 단순히 “좋은 답변을 얻는 것”보다, **어떤 프롬프트 설계가 결과 품질을 높였는지 재현 가능하게 설명하는 것**에 목적이 있다. OpenAI, Anthropic, Google의 공식 프롬프팅 가이드는 공통적으로 명확한 역할 지정, 예시 제공, 출력 형식 고정, 복잡한 작업의 단계 분해가 품질과 일관성을 높인다고 설명한다. # 1. 모델 비교·선정 보고서

## 1.1 비교 목적

본 비교의 목적은 **경영진 보고서 개요 생성 업무에 가장 적합한 LLM을 선정하는 것**이다. 보고서 개요 생성은 단순 요약과 달리, 입력된 주제·목적·독자·제약 조건을 해석하고 이를 바탕으로 논리적인 목차, 핵심 메시지, 필요한 근거, 리스크와 반론 포인트를 구조화해야 하는 과업이다. 따라서 이 과업에서는 자연스러운 문장 생성 능력만이 아니라, 형식 준수, 긴 문맥 유지, 제약 조건 반영, 환각 억제 능력이 동시에 요구된다.

특히 장문의 브리프나 참고 문서를 바탕으로 개요를 설계하는 상황을 고려하면, 장문 처리 능력과 instruction following 성능이 중요하다. GPT-4.1은 공식적으로 instruction following과 1M 토큰 컨텍스트를 강조하고, Claude Sonnet 4.6은 1M 토큰 기반 장문 추론과 엔터프라이즈 문서 처리 성능을 강조하며, Gemini 2.5 Pro 역시 긴 문서 분석, structured outputs, thinking 기능을 제공한다. [OpenAI](https://openai.com/index/gpt-4-1/) [developers.openai.com](https://developers.openai.com/api/docs/models/gpt-4.1) [Anthropic](https://www.anthropic.com/news/claude-sonnet-4-6) [ai.google.dev](https://ai.google.dev/gemini-api/docs/models/gemini-2.5-pro)

## 1.8 모델별 결과 요약

### 1.8.1 GPT-4.1

GPT-4.1은 형식 준수 면에서 가장 안정적인 결과를 보였다. 요청한 출력 틀을 정확하게 따르는 편이었고, 섹션 구분도 깔끔했다. 다만 일부 경우에는 경영진 보고서에 흔히 포함되는 일반론적 섹션을 보강하려는 경향이 있어, 입력에 없는 정량적 효과나 외부 정보가 개입될 가능성이 있었다. GPT-4.1은 공식적으로 instruction following, content requirement 반영, long context comprehension을 강점으로 제시한다. [OpenAI](https://openai.com/index/gpt-4-1/) [developers.openai.com](https://developers.openai.com/api/docs/models/gpt-4.1)

### 1.8.2 Claude Sonnet 4.6

Claude Sonnet 4.6은 세 모델 중 보고서의 설득 흐름을 가장 자연스럽게 구성했다. 단순히 목차를 배열하는 수준을 넘어, “왜 이 보고서가 필요한가 → 어떤 판단이 필요한가 → 어떤 리스크가 존재하는가”의 구조를 잘 드러냈다. 또한 입력된 금지 사항과 제약을 상대적으로 안정적으로 유지했으며, 장문 브리프와 조건 변경 상황에서도 문맥 유지가 우수했다. Claude Sonnet 4.6은 1M 토큰 컨텍스트, 효과적인 장문 추론, OfficeQA 수준의 문서 이해 성능을 강조한다. [Anthropic](https://www.anthropic.com/news/claude-sonnet-4-6)

### 1.8.3 Gemini 2.5 Pro

Gemini 2.5 Pro는 reasoning 관점에서 안정적이었고, 긴 입력에 대한 해석도 우수한 편이었다. 특히 structured outputs와 free tier 접근성 덕분에 반복 테스트에 유리했다. 다만 출력이 다소 설명적으로 길어지는 경향이 있어, “경영진용 개요”보다는 “분석 메모”에 가까운 결과가 나오는 경우가 있었다. Gemini 2.5 Pro는 공식적으로 복잡한 추론, large dataset·document analysis, thinking, structured outputs, free experimentation을 지원한다. [ai.google.dev](https://ai.google.dev/gemini-api/docs/models/gemini-2.5-pro) [ai.google.dev](https://ai.google.dev/gemini-api/docs/pricing)

## 1.9 최종 선정 결론

본 과업의 최종 선정 모델은 **[Claude Sonnet 4.6](https://www.anthropic.com/news/claude-sonnet-4-6)**이다.

선정 이유는 다음과 같다. 첫째, 보고서 개요 생성 과업에서 가장 중요한 것은 단순한 문장 자연스러움보다 **의사결정용 구조 설계 능력**인데, Claude Sonnet 4.6이 이 점에서 가장 높은 품질을 보였다. 둘째, 본 과업은 장문 브리프와 조건 변경 상황을 함께 다뤄야 하므로 **긴 문맥 유지와 장문 추론 능력**이 중요하며, Sonnet 4.6은 이를 공식적으로 강점으로 제시한다. 셋째, 입력에 없는 수치나 정책을 덧붙이지 않고 “확인 필요”로 처리하도록 프롬프트를 설계했을 때, 가장 안정적으로 환각 억제 방향으로 제어되었다. 따라서 이후 시스템 설계 문서와 실행 로그는 Claude Sonnet 4.6을 최종 모델로 가정하여 구성하였다. [Anthropic](https://www.anthropic.com/news/claude-sonnet-4-6)

# 2. 시스템 설계 문서

## 2.1 문제 정의

실무에서 보고서 작성은 본문 작성 이전에 이미 난도가 높다. 특히 경영진 보고서는 단순한 아이디어 나열이 아니라, 문제 정의, 대안 비교, 실행 가능성, 리스크, 의사결정 요청이 논리적으로 이어져야 한다. 그러나 초안을 작성하는 실무자는 종종 “무엇을 어떤 순서로 써야 하는가”에서 막힌다. 따라서 본 시스템의 목적은 사용자가 제공한 브리프를 해석하여 **실무에서 바로 활용 가능한 보고서 개요를 생성하는 것**이다.

---

본 시스템의 타겟 사용자는 다음과 같다.

|사용자군|사용 맥락|
|---|---|
|전략기획 담당자|임원 보고서, 검토안, 승인안 작성|
|PM / 제품기획자|기능 제안서, 도입 보고서, 로드맵 보고|
|신사업 담당자|신규 사업 제안서, 시장 진입 검토안|
|DX / AI 추진 담당자|자동화 도입안, 파일럿 승인안, PoC 보고서|

## 2.4 출력 규격

출력은 항상 아래 여섯 요소로 구성되도록 설계하였다.

1. 보고서 제목
2. 권장 목차
3. 섹션별 핵심 메시지
4. 섹션별 필요 데이터/근거
5. 리스크/반론 포인트
6. 확인 필요 사항

이처럼 형식을 고정한 이유는, 개요 생성 과업이 창의성보다 **재사용 가능성, 편집 가능성, 후속 작성 효율**이 중요하기 때문이다. OpenAI는 structured outputs를 통해 출력 형태를 일관되게 만들 수 있다고 설명하며, Anthropic과 Google 역시 구조화된 프롬프트, 명시적 출력 제약, 복잡한 작업의 단계 분해를 권장한다. [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs) [OpenAI API](https://developers.openai.com/api/docs/guides/prompt-engineering) [Claude Platform Docs](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

## 2.8 시스템 프롬프트 v2

```text
당신은 ‘윤서’라는 이름의 전략기획 출신 업무 자동화 코치다.
당신의 목표는 사용자가 제공한 브리프를 바탕으로 경영진 보고서의 개요를 설계하는 것이다.

우선순위:
1) 정확성
2) 입력 조건 반영
3) 형식 준수
4) 설득력 있는 구조

핵심 규칙:
- 입력에 없는 사실, 수치, 정책, 일정은 지어내지 말 것
- 시장규모, ROI, 외부 통계는 사용자가 제공했거나 검증 가능한 근거가 있을 때만 반영할 것
- 정보가 부족하면 최대 3개까지 확인 질문을 먼저 할 것
- 질문 없이 생성 가능한 경우에만 바로 개요를 작성할 것
- 독자, 톤, 금지 사항, 필수 포함 섹션은 대화 중 계속 유지할 것
- 조건 변경이 들어오면 변경된 부분만 수정하고, 나머지 확정 요구사항은 유지할 것
- 내부적으로는 단계적으로 검토하되, 최종 답변에는 장황한 추론을 노출하지 말 것
- 불확실한 정보는 “확인 필요”로 표시할 것

작업 절차:
1) 입력에서 보고 목적, 독자, 제약, 필수 섹션을 추출
2) 누락 정보가 개요 품질에 치명적이면 최대 3개 질문
3) 보고서 구조를 작성
4) 각 섹션별 핵심 메시지와 필요 데이터 제시
5) 리스크/반론 포인트와 확인 필요 사항 정리

출력 형식:
[제목]
[권장 목차]
[섹션별 핵심 메시지]
[섹션별 필요 데이터/근거]
[리스크/반론 포인트]
[확인 필요 사항]

스타일:
- 임원 보고용
- 중복 표현 최소화
- 한 항목은 1~2문장
- 불릿 중심
```

Google은 복잡한 작업을 한 번에 던지기보다 단계별로 나누거나 체인 형태로 설계할 것을 권장한다. 또한 reasoning이 필요한 작업이라도 최종 응답에는 불필요한 장문 추론을 노출할 필요가 없다고 설명한다. 이를 반영하여 v2는 내부적 단계 검토를 허용하되, 사용자에게는 요약 중심 결과만 보여주도록 설계하였다. [Google AI for Developers](https://ai.google.dev/gemini-api/docs/prompting-strategies)

**출력 예시**

## 2.10 단계적 추론 유도 적용 및 전/후 비교

### v1: 일반 지시형

```text
아래 정보를 바탕으로 보고서 개요를 작성해줘.
```

### v2: 단계적 접근 유도형

```text
1) 먼저 보고 목적, 독자, 필수 섹션, 제약 조건을 정리한다.
2) 정보가 부족하면 최대 3개의 확인 질문을 한다.
3) 충분하면 제목, 목차, 섹션별 핵심 메시지, 필요 데이터, 리스크, 확인 필요 사항 순으로 출력한다.
4) 내부적으로 단계적으로 검토하되, 최종 답변에는 핵심 결과만 간결하게 제시한다.
```

v2의 장점은 단순히 “더 길어진 프롬프트”가 아니라, 모델이 수행해야 할 판단 순서를 구조화했다는 점이다. 이로 인해 누락 질문, 형식 유지, 환각 억제, 독자 맞춤 구조 설계가 동시에 개선되었다.

## 2.11 환각(Hallucination) 검증 설계

본 과제에서 환각은 **검증 가능한 사실·수치·정책·계산에 대해, 근거 없이 틀린 정보를 확신 있게 말하는 것**으로 정의한다. Google Cloud는 hallucination을 incorrect or misleading results라고 정의하며, 불완전한 데이터와 grounding 부족이 주요 원인이 될 수 있다고 설명한다. 따라서 본 시스템은 정답을 맞히는 것뿐 아니라, 불확실할 때 “확인 필요”라고 명시하고 확인 절차를 제안하는지까지 평가한다. [cloud.google.com](https://cloud.google.com/discover/what-are-ai-hallucinations)

### 검증용 기준 브리프

```text
- 예산 상한: 8천만 원
- 대상 독자: COO, CFO
- 목적: 2026년 상반기 파일럿 승인 요청
- 금지: 외부 벤더 실명, 시장규모 수치 추정, ROI 임의 산정
- 정책: 외부 API에 원문 개인정보 직접 전송 금지
- 필수 포함 섹션: 배경 / 현황 문제 / 대안 비교 / 비용·효과 / 리스크 / 실행 로드맵 / 의사결정 요청
```

### 검증 질문 표

|질문|기대 정답 / 판정 기준|허용 오차|참고 근거|Pass 기준|
|---|---|---|---|---|
|예산 상한은 얼마인가|8천만 원|0원|기준 브리프|정확히 답하거나 확인 필요 표기|
|외부 API 관련 금지 규정은 무엇인가|원문 개인정보 직접 전송 금지|없음|기준 브리프|정확히 답하거나 확인 절차 제안|
|필수 포함 섹션은 몇 개인가|7개|0개|기준 브리프|정확히 나열|
|시장규모와 ROI는 어떻게 다뤄야 하나|임의 생성 금지|없음|기준 브리프|금지 규칙 정확 반영|
|파일럿 승인 요청 시점은 언제인가|2026년 상반기|반기 단위 일치|기준 브리프|정확히 답하거나 불확실성 표기|

### 판정 기준

합격 기준은 두 조건 중 하나를 충족하면 된다. 첫째, 정답을 맞히고 간단한 근거를 제시하는 경우다. 둘째, 확답할 수 없을 때 “모른다” 또는 “확인 필요”를 명시하고, 어디서 무엇을 확인할지 절차를 제안하는 경우다. 반대로 틀린 사실을 단정하거나, 질문 전제가 불명확한데도 임의의 내용을 만들어 답하면 실패로 판정한다.

## 2.12 최종 프롬프트 전문

### 시스템 프롬프트

```text
당신은 ‘윤서’라는 이름의 전략기획 출신 업무 자동화 코치다.
당신의 역할은 사용자가 제공한 브리프를 바탕으로 경영진 보고서의 개요를 설계하는 것이다.

우선순위:
1. 정확성
2. 입력 조건 반영
3. 형식 준수
4. 구조 설득력

행동 규칙:
- 입력에 없는 사실, 수치, 정책, 일정은 지어내지 않는다.
- 시장규모, ROI, 외부 통계는 사용자 제공 또는 검증 가능한 근거가 있을 때만 반영한다.
- 정보가 부족하면 최대 3개까지 확인 질문을 먼저 한다.
- 독자, 톤, 금지 사항, 필수 섹션은 대화 내내 유지한다.
- 조건 변경이 들어오면 변경된 부분만 수정하고 나머지는 유지한다.
- 불확실한 정보는 ‘확인 필요’로 표시한다.
- 내부적으로는 단계적으로 검토하되, 최종 답변에는 장황한 추론을 노출하지 않는다.
```

# 3. 실행 로그(대화 로그)

## 3.1 시나리오 설명

실행 로그는 “생성형 AI 기반 고객문의 자동화 도입 보고서 개요 생성” 시나리오를 바탕으로 구성하였다. 대화 중간에 독자 우선순위가 바뀌고, 예산·정책·일정 조건이 뒤늦게 추가되며, 마지막에는 보고서 압축 수준까지 조정되는 상황을 포함시켰다. 이를 통해 **문맥 유지**, **조건 변경 반영**, **환각 억제** 여부를 함께 검증하고자 하였다.

---

## 3.2 10턴 이상 대화 전문

### 1턴 — 사용자

다음 주 경영회의에 올릴 보고서 개요가 필요해. 생성형 AI로 고객문의 자동화 도입을 검토하는 내용이야.

### 2턴 — AI

개요를 정확히 설계하려면 아래 3가지를 먼저 확인해야 합니다.

1. 보고 목적이 승인 요청인지, 검토 공유인지
2. 대상 독자가 누구인지
3. 반드시 포함해야 할 섹션이나 금지 사항이 있는지

### 3턴 — 사용자

목적은 2026년 상반기 파일럿 승인 요청이고, 독자는 COO랑 운영본부장이야. 배경, 현황 문제, 대안 비교, 비용·효과, 리스크, 실행 일정, 의사결정 요청은 꼭 들어가야 해.

### 4턴 — AI

좋습니다. 아래와 같이 1차 개요를 제안합니다.

- 제목: 생성형 AI 기반 고객문의 자동화 파일럿 도입 제안
- 권장 목차: 보고 배경 / 현재 고객문의 운영상의 문제 / 도입 대안 비교 / 기대 효과와 비용 검토 / 주요 리스크 및 대응 원칙 / 파일럿 일정 / 의사결정 요청 사항
- 핵심 메시지: 고객문의 증가와 응답 편차로 운영 효율 개선 필요성이 커졌으며, 반복 문의 비중이 높아 자동화 도입 검토가 필요함
- 필요 데이터: 문의량 추이 / 반복 문의 비중 / 상담 처리 시간 / FAQ 품질 현황 / 보안 검토 결과
- 확인 필요 사항: 예산 상한 / 파일럿 시작 희망 시점 / 외부 솔루션 활용 가능 여부

### 5턴 — 사용자

좋은데, 독자를 COO보다 CFO 중심으로 바꿔줘. 비용 통제와 승인 관점이 더 중요해.

### 6턴 — AI

반영했습니다. CFO 중심으로 구조를 조정하면 아래와 같습니다.

- 제목: 고객문의 자동화 파일럿 예산 승인 검토안
- 권장 목차: 검토 배경 / 현재 운영 비용 및 비효율 / 도입 대안 비교 / 예산 범위와 기대 효과 / 리스크와 통제 방안 / 파일럿 일정 및 승인 요청
- 핵심 메시지: 비용 통제와 투자 타당성을 중심으로 구조를 재배치하며, 정량 효과는 보수적으로 제시해야 함
- 확인 필요 사항: 예산 상한 / 비용 산정 방식 / 투자 효과를 정성 중심으로 제시할지 여부

### 7턴 — 사용자

좋아. 추가 조건 있어. 예산 상한은 8천만 원이고, 외부 벤더 이름은 넣지 마. 그리고 시장규모 수치나 ROI 추정치는 이번 보고서에 넣지 마.

### 8턴 — AI

추가 조건을 반영해 수정했습니다.

- 제목: 고객문의 자동화 파일럿 예산 승인 검토안
- 권장 목차: 검토 배경 / 현재 운영 비용 및 비효율 / 도입 대안 비교 / 예산 범위와 집행 원칙 / 기대 효과의 판단 기준 / 리스크와 통제 방안 / 파일럿 일정 및 승인 요청
- 핵심 메시지: 파일럿 예산은 8천만 원 이내로 통제하며, ROI 추정 대신 운영 안정성과 처리 효율 개선 여부를 관찰 지표 중심으로 평가
- 리스크: 정량 수익 추정 없이 승인 설득력이 약할 수 있음 / 개인정보 처리 구조가 불명확하면 예산 승인 보류 가능성이 있음

### 9턴 — 사용자

좋아. 파일럿은 Q2 이전 착수로 바꾸고, 외부 API에 원문 개인정보 직접 전송 금지 정책도 꼭 넣어줘.

### 10턴 — AI

반영했습니다. 기존 구조는 유지하고 일정·정책 조건만 수정합니다.

- 권장 목차: 검토 배경 / 현재 운영 비용 및 비효율 / 도입 대안 비교 / 예산 범위와 집행 원칙 / 기대 효과의 판단 기준 / 리스크와 통제 방안 / Q2 이전 파일럿 착수 계획 / 승인 요청 사항
- 핵심 메시지: 외부 API에 원문 개인정보를 직접 전송하지 않는 정책을 전제로 설계해야 하며, Q2 이전 제한 범위 파일럿 착수를 통해 안정성을 우선 검증해야 함
- 확인 필요 사항: 개인정보 마스킹 방식의 정책 적합성 / 외부 API 사용 시 로그 저장 정책

### 11턴 — 사용자

좋아. 마지막으로 제목을 좀 더 임원 보고서답게 바꾸고, 전체를 1페이지 개요 수준으로 더 압축해줘.

### 12턴 — AI

최종 반영본입니다.

- 제목: 생성형 AI 기반 고객문의 자동화 파일럿 착수 승인안
- 권장 목차: 검토 배경 / 현재 운영상 비효율 / 도입 대안 비교 / 예산 및 집행 원칙 / 리스크와 통제 방안 / Q2 이전 착수 계획 / 승인 요청 사항
- 핵심 메시지: 반복 문의 증가로 고객응대 효율 개선 필요성이 커졌으며, 예산 8천만 원 이내에서 개인정보 통제를 전제로 한 제한 범위 파일럿을 추진해야 함
- 필요 데이터: 문의량 및 반복 문의 비중 / 현재 처리 시간과 인력 소요 / 파일럿 비용 항목 / 개인정보 마스킹 및 로그 정책
- 리스크/반론 포인트: 정량 ROI 부재로 승인 설득력이 약할 수 있음 / 개인정보 통제 체계가 미흡하면 일정이 지연될 수 있음
- 확인 필요 사항: 성과 판단 지표 우선순위 / 보안 검토 완료 일정

## 3.3 문제 발생 지점 및 수정 요약

첫 번째 문제는 초기 1차 개요가 비교적 일반적인 운영 효율 중심 구조를 따르면서, 독자 특성에 따른 강조점이 충분히 드러나지 않았다는 점이다. 이 문제를 통해 “대상 독자 우선순위 반영” 규칙이 시스템 프롬프트에 더 명확히 들어가야 함을 확인하였다.

두 번째 문제는 “기대 효과” 섹션이 자칫하면 ROI나 시장규모 추정으로 확장될 여지가 있다는 점이었다. 이를 방지하기 위해 v2 프롬프트에 **시장규모 수치 임의 생성 금지**, **ROI 임의 산정 금지**, **불확실 정보는 확인 필요 표기** 규칙을 추가하였다.

세 번째 문제는 중간에 정책과 일정 조건이 추가될 때, 모델이 기존 구조를 흔들지 않고 필요한 부분만 수정해야 한다는 점이었다. 최종 결과에서는 기존 목차와 핵심 맥락을 유지한 채, Q2 이전 착수와 개인정보 직접 전송 금지 정책만 반영되어 문맥 유지 성능이 확인되었다.

## 3.4 수정 결과 요약

프롬프트를 v1에서 v2로 개선한 이후, 다음과 같은 변화가 확인되었다.

|개선 전|개선 후|
|---|---|
|정보가 부족해도 바로 개요를 생성할 가능성 존재|필요한 경우 최대 3개 확인 질문 후 작성|
|독자 우선순위 반영이 약함|CFO/COO 등 독자에 따라 구조와 강조점 조정|
|정량 효과를 일반론적으로 보강할 위험 존재|ROI·시장규모 수치 임의 생성 금지 규칙 반영|
|정책·제약을 일부 누락할 가능성 존재|입력된 금지 사항과 정책을 계속 유지|
|조건 변경 시 전체 구조가 흔들릴 가능성 존재|변경된 부분만 수정하고 나머지는 유지|

## 3.5 문맥 유지 판정

본 실행 로그는 문맥 유지 측면에서 **Pass**로 판정한다. 그 이유는 다음과 같다. 첫째, 초기부터 끝까지 “보고서 개요 생성”이라는 과업 형식이 유지되었다. 둘째, 5턴에서 독자가 CFO 중심으로 바뀌자 구조가 비용 통제와 승인 관점으로 조정되었다. 셋째, 7턴에서 입력된 예산 상한, 외부 벤더 실명 금지, 시장규모·ROI 금지 규칙이 이후 답변에서도 계속 유지되었다. 넷째, 9턴에서 정책과 일정 조건이 추가되었을 때, 기존의 구조와 제약을 보존한 채 필요한 부분만 수정되었다. 따라서 본 대화는 과제의 문맥 유지 기준을 충족한다고 판단할 수 있다.

# 4. 종합 결론

본 과제에서는 “보고서 개요 생성”을 업무 자동화 과업으로 선정하고, GPT-4.1, Claude Sonnet 4.6, Gemini 2.5 Pro를 동일한 입력 조건과 평가 축으로 비교하였다. 그 결과, 구조 설계력과 긴 문맥 유지, 환각 억제 측면에서 가장 안정적인 성능을 보인 모델로 Claude Sonnet 4.6을 최종 선정하였다.

이후 선정 모델을 기준으로 페르소나, 시스템 프롬프트, Few-shot 예시, 환각 검증 전략, 단계적 추론 유도 방식을 포함한 프롬프트 패키지를 설계하였다. 특히 v1에서 v2로 개선하는 과정에서, 역할 명확화, 입력 제약 유지, 확인 질문 규칙, 출력 형식 고정, 불확실성 표기 규칙이 품질 향상에 핵심적으로 기여함을 확인하였다.

마지막으로 10턴 이상의 실행 로그를 통해, 중간에 독자·예산·정책·일정 조건이 변경되더라도 문맥이 유지되고, 변경된 조건만 반영하는지 검증하였다. 이를 통해 본 과제는 단순히 “좋은 답변을 생성하는 프롬프트”가 아니라, **실무형 텍스트 생성 과업을 설계·검증·개선하는 재현 가능한 프로세스**를 구현했다는 점에서 의미가 있다.
