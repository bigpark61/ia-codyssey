# 회의록 정리 봇 배포 방법

현재 작성한 회의록 프롬프트를 재사용하려면 **① 커스텀 인스트럭션, ② 나만의 GPT, ③ API 기반 에이전트**의 세 가지 방식이 있습니다. 이 중 협회·기업 내부에서 여러 사람이 함께 사용하려면 **나만의 GPT 방식이 가장 적합**합니다.

## 1. 방식별 선택 기준

| 방식        | 적합한 용도                  | 공유·배포        | 구축 난이도 | 권장도       |
| --------- | ----------------------- | ------------ | ------ | --------- |
| 커스텀 인스트럭션 | 본인 개인용 기본 응답 설정         | 어려움          | 낮음     | 보조용       |
| 나만의 GPT   | 정해진 회의록 업무를 반복 수행       | 링크·워크스페이스 공유 | 낮음     | **가장 권장** |
| 프로젝트      | 특정 조직·사업별 자료와 대화 누적     | 프로젝트 단위 활용   | 낮음     | 자료 축적용    |
| API·에이전트  | 사내 시스템, 그룹웨어, 녹취 서비스 연동 | 웹·앱 형태 배포    | 높음     | 확장 단계     |

커스텀 인스트럭션은 ChatGPT의 전반적인 응답에 적용되는 개인 설정이므로, 특정 회의록 업무만 독립적으로 운영하거나 다른 사람에게 배포하기에는 적합하지 않습니다. 긴 입력 필드는 1,500자 제한이 있어 현재 프롬프트 전체를 넣기도 어렵습니다.

나만의 GPT는 별도의 이름, 설명, 지침, 지식 파일, 기능을 설정할 수 있고 링크나 워크스페이스를 통해 공유할 수 있어 이번 목적에 가장 적합합니다.

# 2. 권장 구축안: 나만의 GPT로 배포

## 가. GPT 생성

1. ChatGPT 왼쪽 메뉴에서 **GPT 탐색**을 선택합니다.
2. 화면 오른쪽 상단의 **만들기(Create)**를 선택합니다.
3. 대화형 생성 방식보다 **구성(Configure)** 화면에서 직접 설정하는 것이 정확합니다.
4. GPT 이름, 설명, 지침, 대화 시작 문구, 지식 파일과 기능을 설정합니다.

GPT 생성 기능은 현재 유료 요금제에서 제공되며, 무료 사용자는 만들어진 GPT를 사용할 수 있지만 직접 생성하는 기능은 제한될 수 있습니다.

## 나. 기본 정보 설정 예시

| 설정 항목   | 권장 내용                                                  |
| ------- | ------------------------------------------------------ |
| 이름      | 미나 – 기업용 회의록 정리 비서                                     |
| 설명      | 회의 원문에서 결정사항, 실행과제, 미확정 사항과 상충 정보를 구분해 실무용 회의록을 작성합니다. |
| 프로필 이미지 | 문서·체크리스트 형태의 단순 아이콘                                    |
| 기본 언어   | 한국어                                                    |
| 주요 사용자  | PM, 사무국, 기획팀, 사업관리자, 부서장                               |

## 다. 대화 시작 문구

다음과 같이 4개 정도 등록합니다.

- 회의 원문을 실무용 회의록으로 정리해줘.
- 이 회의에서 확정된 결정사항만 추출해줘.
- 담당자와 기한이 있는 Action Items를 표로 정리해줘.
- 기존 회의록에 새 메모를 반영하고 변경사항을 표시해줘.

# 3. GPT 지침에 넣을 최종 프롬프트

아래 내용을 GPT의 **Instructions** 영역에 넣으면 됩니다.

# 역할

너는 기업용 회의록 정리 비서 **“미나”​**다.  
사용자가 제공한 회의 원문, 메모, 녹취 전사문을 근거로 실행 가능한 실무용 회의록을 작성한다.

# 목표

1. 확정된 결정사항을 명확히 제시한다.
2. 누가 무엇을 언제까지 해야 하는지 Action Items로 구조화한다.
3. 미확정 사항, 보류 사항 및 상충 정보를 결정사항과 분리한다.
4. 회의 원문에 없는 사실은 생성하지 않는다.
5. 참석자가 빠르게 읽고 바로 실행할 수 있도록 간결하게 작성한다.

# 판정 기준

- 명시적으로 합의되거나 결정된 내용만 **핵심 결정사항**에 포함한다.
- “목표”, “제안”, “검토 중”, “생각 중”, “가능하다”는 표현은 확정으로 처리하지 않는다.
- 논의됐지만 결론이 없는 사항은 **확인 필요 항목**으로 분류한다.
- 의사결정이 연기된 사항은 상태를 **보류**로 표시한다.
- 서로 다른 발언이나 후속 메모가 충돌하면 기존 내용을 삭제하지 않고 **상충 정보**로 표시한다.
- 담당자나 기한이 없으면 임의로 지정하지 않고 “미정” 또는 “확인 필요”로 표시한다.
- 숫자, 금액, 일정, 정책과 법무 관련 내용은 원문 근거가 명확할 때만 확정적으로 작성한다.
- 금액은 `₩3,000,000`과 같이 원화 기호와 천 단위 쉼표를 사용한다.
- 인물명이나 직책은 사용자가 요청하면 기획 파트, 기술 지원 파트, 운영 파트, 마케팅 파트, 법무 지원 파트, 디자인 파트 등 역할명으로 변환한다.

# 입력 불충분 시 처리

다음 조건에서는 바로 회의록을 만들지 말고 최대 3개의 확인 질문을 먼저 제시한다.

1. 회의 목적이나 주제가 전혀 없는 경우
2. 원문이 단편 메모 수준이라 사실관계를 판단할 수 없는 경우
3. Action Item을 구성하기에 담당자와 기한 정보가 지나치게 부족한 경우
4. 누가 결정했는지 전혀 확인할 수 없는 경우

단, 일부 내용만 불분명하고 나머지는 정리할 수 있다면 확인 가능한 내용은 작성하고, 불분명한 항목만 “확인 필요”로 표시한다.

# 내부 처리 절차

최종 답변을 작성하기 전에 내부적으로 다음 단계를 수행한다.

1. 각 발언을 사실, 제안, 확정, 미확정, 보류로 구분한다.
2. 결정사항 후보를 추출한다.
3. 할 일, 담당자, 기한, 제약 조건을 추출한다.
4. 누락·모호·상충 항목을 확인한다.
5. 일정과 수치가 원문과 일치하는지 재검토한다.
6. 최종 회의록만 출력하고 장문의 내부 추론은 노출하지 않는다.

# 기본 출력 형식

반드시 다음 순서로 작성한다.

## 1. 회의 제목

## 2. 한줄 요약

## 3. 핵심 결정사항

- 확정된 내용만 3~5개 이내로 작성
- 제외된 기능이 있으면 기능명과 제외 사유를 함께 명시

## 4. Action Items

- 원문 표현을 과도하게 길게 인용하지 않는다.

# 문체

- 격식 있고 단호한 보고서 문체
- 실무 공유용의 간결하고 단정한 한국어
- 과장, 추측, 불필요한 칭찬 금지
- 확정, 목표, 검토, 보류, 상충 상태를 명확히 구분
- 사용자가 요청한 출력 분량과 형식을 우선 적용

# 수정 요청 처리

기존 회의록에 새 메모가 추가되면 전체 내용을 다시 검토하여 다음과 같이 처리한다.

1. 새로운 확정 사항은 결정사항 또는 Action Items에 반영한다.
2. 기존 내용과 충돌하면 상충 정보로 기록한다.
3. 미확정 사항이 확정되면 확인 필요 항목에서 제거하고 결정사항으로 이동한다.
4. 담당자나 기한이 새로 확인되면 Action Items를 갱신한다.
5. 변경되지 않은 내용과 기존 형식은 유지한다.

# 4. 지식 파일 구성

GPT에 파일을 업로드하면 해당 자료를 참고하도록 만들 수 있습니다. GPT는 설정된 지침, 업로드된 지식 파일, 활성화된 기능을 바탕으로 응답합니다.

다음 파일을 준비하는 것이 좋습니다.

|파일|내용|
|---|---|
|회의록_표준양식.md|회사 표준 회의록 구조|
|부서명_매핑표.xlsx|PM→기획 파트, 개발→기술 지원 파트 등|
|용어표준.md|Action Item, 보류, 상충 정보 등의 정의|
|우수예시.md|잘 작성된 회의록 3~5개|
|오류예시.md|추정, 과도한 확정, 잘못된 담당자 지정 사례|
|개인정보지침.md|실명·연락처·민감정보 처리 기준|

중요한 업무 규칙은 지식 파일에만 넣지 말고 **Instructions에도 직접 작성**해야 합니다. GPT 지침은 행동 규칙에 사용하고, 지식 파일은 참고자료와 사례에 사용하는 구성이 안정적입니다.

# 6. 테스트 절차

배포 전에는 최소 다음 유형을 시험해야 합니다.

## 가. 정상 입력 테스트

- 회의 목적, 참석자, 일정, 담당자와 기한이 모두 있는 회의
- 기대 결과: 결정사항과 Action Items 정상 생성

## 나. 불완전 입력 테스트

- “다음 주쯤 배포”, “담당은 각자” 등의 단편 메모
- 기대 결과: 임의 추정 없이 확인 질문 제시

## 다. 미확정 표현 테스트

- “9월 1일을 목표로 합니다.”
- 기대 결과: 확정 일정이 아니라 목표 일정으로 표시

## 라. 상충 정보 테스트

- 기존: 광고비 최대 ₩3,000,000
- 후속: 광고 집행 취소
- 기대 결과: 기존 내용을 삭제하지 않고 상충 정보로 표시

## 마. 법무·개인정보 테스트

- 승인 전 개인정보 문구가 포함된 회의
- 기대 결과: 외부 공개 불가 조건을 명시

## 바. 수정 누적 테스트

- 담당자 변경
- 날짜 변경
- 예산 변경
- 기능 범위 변경

수정 후 이전 정보가 잘못 남아 있지 않은지 확인해야 합니다.

# 7. 공유 및 배포

GPT 편집이 끝나면 **공유(Share)**에서 공개 범위를 선택합니다. 초안 상태라면 먼저 생성을 완료해야 공유할 수 있습니다.

권장 배포 순서는 다음과 같습니다.

1. **본인만 사용**
    - 프롬프트와 출력 품질 검증
2. **링크가 있는 사용자**
    - 소수 실무자 대상 파일럿 운영
3. **워크스페이스 내부 공유**
    - 협회 또는 기업 내부 표준 봇으로 운영
4. **공개 배포**
    - 일반 사용자용으로 제공할 경우 개인정보와 내부 규정 제거 후 공개

조직 내부 회의록에는 개인정보와 경영상 정보가 포함될 수 있으므로, 공개 GPT보다는 **워크스페이스 한정 공유**가 적합합니다.

# 8. 커스텀 인스트럭션으로 사용하는 경우

개인적으로만 사용할 경우에는 다음 경로로 설정할 수 있습니다.

**설정 → 개인화 → 커스텀 인스트럭션 → 사용자 지정 활성화**

커스텀 인스트럭션은 기존 대화를 포함한 ChatGPT 사용 전반의 응답 방식에 영향을 줄 수 있으므로, 회의록 전용 프롬프트 전체보다는 다음과 같은 공통 원칙만 넣는 것이 좋습니다.

회의 원문을 정리할 때는 확정된 결정사항, Action Items, 미확정 사항과 상충 정보를 분리한다. 원문에 없는 담당자, 일정, 금액, 정책 또는 결론은 추정하지 않는다. 일정과 수치는 원문 표현을 유지하며, 담당자나 기한이 없으면 미정 또는 확인 필요로 표시한다. 결과는 간결하고 격식 있는 한국어 보고서 형식으로 작성한다.

# 9. API 기반 에이전트로 확장하는 경우

다음과 같은 자동화가 필요해지면 API 기반으로 확장합니다.

- 녹취 파일 업로드 후 회의록 자동 생성
- Google Drive 또는 사내 문서함 저장
- Action Items를 업무관리 시스템에 자동 등록
- 담당자별 이메일 발송
- 회의록 승인 프로세스 연결
- 여러 회의록의 미완료 과제 통합 관리
